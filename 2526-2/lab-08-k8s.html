Tổng quan & Cài đặt

Bài toán thực tế

  • 2003: Google vận hành hàng trăm nghìn server
  • Search phục vụ người dùng toàn cầu
  • Hạ tầng tăng nhanh, dịch vụ ngày càng nhiều
  • Quản lý thủ công không còn khả thi
    • Deploy từng máy qua SSH
    • Lỗi → tự phát hiện, tự restart
    • Traffic tăng → tự scale
    • Khó nhìn trạng thái toàn hệ thống
Urs Hölzle — Phó giám đốc công nghệ Google (2003)

Với tốc độ mở rộng hiện tại, hệ thống của chúng ta sẽ sụp đổ nghiêm trọng vào cuối năm 2004.

Hệ thống càng lớn → không thể quản lý thủ công từng thành phần.

Lịch sử phát triển

  • ~2003: Google phát triển hệ thống Borg để quản lý hạ tầng quy mô lớn.
  • Sau đó: Omega ra đời, kế thừa và cải tiến từ Borg.
  • Tiên phong các cơ chế tự động hóa: deploy • scale • self-healing.
  • 2014: Open-source dự án với tên Kubernetes.
  • Hiện nay: Được sử dụng rộng rãi và gần như là tiêu chuẩn trong Cloud-native.
Kubernetes là kết quả của hơn 10 năm kinh nghiệm vận hành hạ tầng quy mô lớn tại Google, được chuẩn hóa và phổ biến cho cộng đồng.

Kubernetes

Là hệ thống điều phối container giúp tự động hóa việc triển khai, mở rộng và quản lý các ứng dụng container trên một cụm máy chủ.

  • Deploy: Tự động triển khai ứng dụng lên các cụm máy chủ phù hợp nhất.
  • Scale: Tăng hoặc giảm số lượng bản sao ứng dụng tức thì theo tải thực tế.
  • Self-healing: Tự động khôi phục, khởi động lại hoặc thay thế container khi có lỗi.
  • K8s: là viết tắt của Kubernetes, với 8 chữ cái ở giữa chữ "K" và chữ "s".

Kubernetes giống như một hệ điều hành cho cụm máy chủ container

2.
Kiến trúc Kubernetes

Thành phần chính

Control Plane

  • api server: Tiếp nhận yêu cầu
  • scheduler: Điều phối vị trí Pod
  • controller manager: Duy trì trạng thái
  • etcd: Kho lưu trữ dữ liệu cấu hình

Worker Node

  • kubelet: Quản trị viên tại chỗ của Node
  • kube-proxy: Quản lý kết nối mạng nội bộ
  • container runtime: Môi trường chạy Container

Sơ đồ kiến trúc chi tiết các thành phần Kubernetes

Api Server

  • Là cửa chính của Kubernetes
  • Nhận request từ:
    • kubectl (dòng lệnh)
    • Giao diện quản lý (Dashboard)
    • Hệ thống tự động triển khai
  • Kiểm tra quyền truy cập
  • Lưu và lấy trạng thái hệ thống qua etcd
User / Admin ➔ [ API Server ] ➔ etcd
Nguyên lý thiết kế: Các thành phần trong K8s không bao giờ nói chuyện trực tiếp với nhau. Mọi luồng giao tiếp và điều khiển đều bắt buộc phải đi qua API Server.

etcd

  • Cơ sở dữ liệu key-value: lưu toàn bộ trạng thái của cluster (Pod, Node, Service, cấu hình,...)
  • Hệ thống phân tán: chạy trên nhiều node để tăng độ tin cậy và khả năng chịu lỗi
  • Leader – Follower (Raft): chỉ leader xử lý ghi dữ liệu, các node khác đồng bộ theo
  • Tự phục hồi: khi leader lỗi, các node còn lại tự bầu leader mới

API Server đọc và ghi dữ liệu tại etcd

etcd luôn lưu trạng thái hiện tại của cluster

Scheduler

  • Chịu trách nhiệm chọn Node để chạy Pod
  • Dựa vào nhiều yếu tố:
    • Tài nguyên: CPU, RAM còn trống
    • Ràng buộc: label, selector
    • Quy tắc: nodeAffinity, podAffinity
  • Không chạy container, chỉ quyết định vị trí

Quy trình xử lý:

  1. Lắng nghe Pod chưa được gán Node
  2. Tìm các Node phù hợp
  3. Chọn Node tốt nhất
  4. Ghi kết quả về API Server
API Server ➔ Scheduler ➔ chọn Node ➔ API Server ➔ kubelet chạy Pod
Scheduler = “Người phân công công việc”

Controller Manager

  • Chạy các controller để duy trì trạng thái hệ thống ổn định.
  • Ví dụ:
    • Đảm bảo số lượng Pod luôn đúng như mong muốn.
    • Tạo lại Pod khi bị lỗi hoặc bị xóa.
  • Liên tục so sánh giữa trạng thái mong muốn và trạng thái thực tế.
desired state ➔ controller ➔ điều chỉnh ➔ actual state
Controller = "Người giám sát và sửa lỗi tự động"

kubelet

  • Agent quản lý: Là thành phần chạy trên mọi Worker Node trong Cluster.
  • Tiếp nhận lệnh: Theo dõi các Pod được API Server gán cho Node của mình.
  • Vận hành Container: Ra lệnh cho container runtime tạo hoặc quản lý Container.
  • Giám sát: Kiểm tra sức khỏe Container và tự động khởi động lại nếu gặp lỗi.
API Server ➔ kubelet ➔ container runtime ➔ chạy Pod
kubelet = "Người thực thi tại Node"

Container Runtime

  • Động cơ thực thi: Thành phần thực sự chạy các Container trên Node.
  • Làm việc qua kubelet: Nhận chỉ thị từ kubelet để thực hiện các thao tác kỹ thuật:
    • Pull image từ Registry (Docker Hub, v.v.).
    • Khởi tạo, chạy, dừng hoặc xóa container.
  • Các chuẩn phổ biến: containerd (phổ biến nhất), CRI-O.
kubelet ➔ container runtime ➔ [Container thực thi]
Kubernetes không tự chạy container, nó điều phối thông qua Runtime

kube-proxy

  • Quản lý mạng: Là thành phần thiết lập và duy trì các luật mạng trên mỗi Node.
  • Định tuyến Traffic: Đảm bảo các yêu cầu từ Service được chuyển chính xác đến các Pod tương ứng.
  • Cân bằng tải: Tự động phân phối traffic giữa các bản sao của Pod để tối ưu hiệu suất.
  • Công nghệ hỗ trợ: Sử dụng iptables hoặc IPVS để điều hướng dữ liệu ở tầng mạng.
Client ➔ Service ➔ kube-proxy ➔ Pod (đích)
kube-proxy = "Người điều phối Traffic trong Cluster"

Luồng hoạt động

  1. User ➔ kubectl ➔ API Server
  2. API Server ➔ xác thực ➔ ghi trạng thái vào etcd
  3. Controller ➔ phát hiện trạng thái mới và tạo Pod (nếu cần)
  4. Scheduler ➔ chọn Node phù hợp cho Pod
  5. API Server ➔ cập nhật Pod đã được gán Node
  6. kubelet ➔ theo dõi Pod và yêu cầu container runtime chạy
  7. Container Runtime ➔ pull image và chạy container
  8. kube-proxy ➔ định tuyến traffic từ Service đến Pod
  9. Controller ➔ liên tục kiểm tra và điều chỉnh trạng thái

Kubernetes hoạt động theo vòng lặp:

Lưu trạng thái ➔ Thực thi ➔ Kiểm tra ➔ Điều chỉnh

3.
Các khái niệm cơ bản

Pod

  • Là đơn vị nhỏ nhất trong Kubernetes
  • Chứa 1 hoặc nhiều container chạy cùng nhau
  • Các container trong Pod:
    • Dùng chung mạng (IP)
    • Có thể chia sẻ dữ liệu

Pod là “đơn vị triển khai” của Kubernetes

Deployment

  • Dùng để quản lý và triển khai Pod
  • Hỗ trợ:
    • Số lượng Replica (scale)
    • Rolling update (cập nhật không gián đoạn)
    • Tự động tạo lại Pod khi bị lỗi
  • Không làm việc trực tiếp với container, mà quản lý Pod

Deployment = “cách Kubernetes chạy và quản lý ứng dụng”

Service

  • Cung cấp địa chỉ truy cập ổn định cho các Pod
  • Kết nối:
    • Pod ↔ Pod
    • Pod ↔ bên ngoài
  • Dùng label selector để chọn các Pod phía sau
  • Hỗ trợ load balancing giữa các Pod
  • Các loại chính:
    • ClusterIP (nội bộ cluster)
    • NodePort (mở port trên node)
    • LoadBalancer (ra Internet)
Client → Service → kube-proxy → Pod

Service = “cổng truy cập ổn định cho Pod”

Job / CronJob

  • Dùng cho tác vụ chạy một lần rồi kết thúc
  • Job:
    • Chạy task một lần
    • Đảm bảo hoàn thành thành công
  • CronJob:
    • Chạy task theo lịch (giống cron)
    • Ví dụ: mỗi ngày, mỗi giờ
  • Thường dùng cho:
    • Batch processing
    • Backup dữ liệu
    • Pipeline tự động

Job = “làm xong rồi dừng”, khác với Deployment chạy liên tục

Deploy & Scale

Tạo Deployment:

kubectl create deployment web --image=nginx

Scale lên 3 bản sao:

kubectl scale deployment web --replicas=3

Deployment đảm bảo số lượng Pod luôn đúng (self-healing + scaling)

Expose & Debug

Expose ra bên ngoài:

kubectl expose deployment web --type=NodePort --port=80

Debug hệ thống:


            kubectl get pods
            kubectl logs <pod>
            kubectl exec -it <pod> -- /bin/bash

Service cung cấp địa chỉ truy cập ổn định cho Pod

Storage (PV / PVC)

  • Dữ liệu trong Pod sẽ mất khi Pod bị xóa
  • PV + PVC giúp lưu dữ liệu lâu dài
  • Dùng cho:
    • Database
    • Model checkpoint

Pod chết → dữ liệu vẫn còn

Mapping sang AI system

  • Pod → chạy training hoặc task nhỏ
  • Deployment → service inference (API model)
  • Job → pipeline training / batch job
  • PV → dataset / checkpoint

Kubernetes = nền tảng chạy hệ thống AI end-to-end
