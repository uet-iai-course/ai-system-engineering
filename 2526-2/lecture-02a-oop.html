0.
Hạn chế của ADT &
Động lực OOP

Ôn lại: ADT (Bài 4)

✓ ADT giải quyết được
  • Ẩn biểu diễn nội tại
  • Tách giao diện khỏi cài đặt
  • Thay thế cài đặt tự do
✗ ADT không có
  • Tái dùng mã giữa các kiểu
  • Tương thích kiểu (lưu chung)
  • Điều phối đúng theo kiểu thực tế
Bài này
Lần lượt giải quyết từng thiếu sót — mỗi thiếu sót dẫn đến một trụ cột OOP

Counter — một ADT quen thuộc


class Counter {
public:
    void reset() { value = 0; }      // giao diện
    int  get()   { return value; }   //   (signature
    void inc()   { value++; }        //    + operations)
private:
    int value = 0;                   // biểu diễn — thực sự ẩn
};

// Client chỉ nhìn thấy giao diện — value không truy cập được:
Counter c;
c.inc();  c.inc();  c.inc();
c.get();  // → 3     // c.value = 99; → LỖI biên dịch ✓

Counter hoạt động tốt — nhưng điều gì xảy ra khi ta cần mở rộng nó?

Bài toán mở rộng

Yêu cầu
Tạo NewCounter: hoạt động như Counter, nhưng còn đếm thêm số lần reset() được gọi.
  • Không được sửa Counter gốc
  • get() và inc() giữ nguyên — chỉ reset() thêm logic
  • Lý tưởng: dùng NewCounter ở mọi nơi đang dùng Counter — không phá vỡ code cũ

ADT thuần không có cơ chế cho điều này → cần mở rộng mô hình

Cách 1: Sao chép toàn bộ


class NewCounter1 {
public:
    void reset() { value = 0; numReset++; } // ← thêm logic đếm reset
    int  get()   { return value; }   // ← chép từ Counter
    void inc()   { value++; }        // ← chép từ Counter
private:
    int value    = 0;
    int numReset = 0;
};
  • ✖ Dư thừa mã — get, inc giống hệt Counter
  • ✖ Bảo trì khó — sửa lỗi Counter gốc phải sửa tay ở mọi bản sao

Cách 2: Nhúng Counter


class NewCounter2 {
public:
    void reset() { inner.reset(); numReset++; } // ← dùng Counter, thêm logic đếm reset
    int  get()   { return inner.get(); }  // ← dùng Counter
    void inc()   { inner.inc(); }         // ← dùng Counter
private:
    Counter inner;
    int     numReset = 0;
};
  • ✖ Không tương thích kiểu — không lưu chung Counter[] được
  • ✖ Điều phối tĩnh — dù ép tương thích, vẫn gọi sai phương thức

Tương thích & Điều phối sai


Counter arr[10];
NewCounter2 nc;
arr[0] = nc;    // ① LỖI biên dịch: NewCounter2 không phải Counter

// Giả sử ép con trỏ cho "tương thích":
Counter* p = reinterpret_cast<Counter*>(&nc);
p->reset();     // ② Counter::reset() được gọi
                //   numReset KHÔNG bao giờ tăng!
  • Kiểu con trỏ tĩnh là Counter* → trình biên dịch gọi Counter::reset()
  • Cần điều phối động: chọn phương thức theo kiểu thực tế lúc chạy

Bốn trụ cột OOP

Ví dụ Counter ADT dẫn đến Kế thừa và Đa hình. Cùng với Đóng gói và Trừu tượng hóa, OOP có 4 trụ cột kinh điển:

Đóng gói (Encapsulation)
Data + phương thức gói trong một đối tượng.
→ Object, Class, public/private
Kế thừa (Inheritance)
Tái dùng + mở rộng lớp cha — không chép mã
→ Kiểu con, Ghi đè
Trừu tượng hóa (Abstraction)
Tách giao diện khỏi cài đặt — ẩn độ phức tạp
→ Lớp trừu tượng, Interface
Đa hình (Polymorphism)
Một giao diện, nhiều hành vi — chọn theo kiểu thực tế
→ Điều phối động, Liên kết muộn

1.
Đóng gói
(Encapsulation)

Đặt vấn đề


struct Counter {   // struct: mọi thành viên public mặc định
    int value = 0; // ← PUBLIC — ai cũng ghi được!
    void reset() { value = 0; }
    int  get()   { return value; }
    void inc()   { value++; }
};
Counter c;
c.inc();  c.inc(); // value = 2
c.value = -99;     // hợp lệ — không lỗi biên dịch!
c.get();           // → -99
  • Bất biến bị vi phạm: value < 0 — không có gì ngăn cản
  • Giải pháp: value chỉ được sửa qua reset() và inc()

Đóng gói (Encapsulation)

Định nghĩa
Gom dữ liệu và phương thức vào một đơn vị (đối tượng), và kiểm soát quyền truy cập từ bên ngoài.
  • Kiểm soát truy cập bằng public / private / protected.
  • Đối tượng = dữ liệu + phương thức — dữ liệu chỉ thay đổi qua phương thức

Đóng gói (Encapsulation)

✗ Không có private

struct Counter {
    int value = 0;  // public!
    // ...
    // ...
};
c.value = -99;  // OK — no error
✓ Có private

class Counter {
private:
    int value = 0;  // hidden
    // ...
};
c.value = -99;  // COMPILE ERROR
  • Bất biến (invariant): điều kiện luôn đúng về trạng thái
  • Counter: value ≥ 0 tại mọi thời điểm
  • private để ngôn ngữ ép bất biến

Đóng gói (Encapsulation)

Các mức độ truy cập:

  • private: chỉ trong lớp — ẩn hoàn toàn
  • public: mọi client — giao diện
  • protected: lớp + lớp con — dùng khi kế thừa

class Counter {
public:
    void reset() { x = 0; }  int get() { return x; }  void inc() { x++; }
protected:
    int x = 0;   // lớp con truy cập được, client thì không
};
class NewCounter : public Counter {
    int nr = 0;
public:
    void reset() override { x = 0; nr++; }  // x là protected, dùng trực tiếp
};

Đối tượng (Object)

Định nghĩa
"Vỏ bọc" (capsule) gói dữ liệu và thao tác trên dữ liệu đó.
  • Dữ liệu: biến thực thể (instance variables)
  • Thao tác: phương thức (methods)
  • Giao tiếp: object.method(args) — this trỏ tới đối tượng nhận
Counter (đối tượng)
x dữ liệu (private)
reset this.x = 0
get return this.x
inc this.x++

Lớp (Class) — Khuôn mẫu tạo Đối tượng


class Counter {
public:
    void reset() { x = 0; }
    int  get()   { return x; }
    void inc()   { x++; }
private:
    int x = 0;
};

Counter c;    // mỗi đối tượng có x riêng
Counter d;
c.inc();      // c.x = 1, d.x vẫn = 0
c
x = 1
d
x = 0

Counter (lớp)

mã dùng chung

Hàm khởi tạo (Constructors)


class Counter {
public:
    Counter()          : value(0)     {} // mặc định
    Counter(int init)  : value(init)  {} // có tham số

    void reset() { value = 0; }
    int  get()   { return value; }
    void inc()   { value++; }
private:
    int value;  // không tự khởi tạo trong C++!
};

Counter c;      // value = 0
Counter d(5);   // value = 5

Hàm khởi tạo (Constructors)

  • Tên constructor = tên lớp, không có kiểu trả về
  • Initializer list (: value(0)) — khởi tạo trước khi thân hàm chạy
  • Đảm bảo bất biến ngay từ lúc tạo — không có đối tượng ở trạng thái không hợp lệ

Cái gì nên public?

✗ Thiết kế xấu

class Counter {
public:
    int value;   // lộ trạng thái ra ngoài
    void reset() { value = 0; }
    void inc()   { value++; }
    void clamp(int lo, int hi); // lộ nội bộ
};
✓ Thiết kế tốt

class Counter {
public:          // giao diện: "làm gì"
    void reset();
    int  get();
    void inc();
private:         // chi tiết: "làm thế nào"
    int  value;
    void clamp(int lo, int hi);
};
  • Nguyên tắc: public = what (hành vi) · private = how (cài đặt)
  • Lợi ích: đổi cài đặt tự do mà client không hay biết

2.
Kế thừa
(Inheritance)

Kế thừa (Inheritance)

Định nghĩa
Lớp con (subclass) nhận toàn bộ thuộc tính và phương thức của lớp cha (superclass), rồi có thể mở rộng hoặc ghi đè theo nhu cầu riêng.
  • Tái sử dụng — không viết lại cài đặt đã có
  • Mở rộng — thêm thuộc tính / phương thức mới vào lớp con
  • Ghi đè — thay thế phương thức lớp cha bằng cài đặt chuyên biệt hơn

Ví dụ: Kế thừa

Counter → NewCounter: hoạt động như Counter, nhưng đếm thêm số lần reset được gọi


class NewCounter : public Counter {
    int num_reset = 0;
public:
    // KẾ THỪA: get() và inc() lấy từ Counter
    // GHI ĐÈ: reset() viết lại
    void reset() override {
        Counter::reset();   // gọi lớp cha
        num_reset++;
    }
    // MỞ RỘNG: thêm phương thức mới
    int howmanyResets() const { return num_reset; }
};

Ghi đè (Method Overriding)

Lớp con định nghĩa lại phương thức virtual của lớp cha:


class Counter {
public:
    virtual void reset() { x = 0; }  // virtual → cho phép lớp con ghi đè
    int  get() { return x; }
    void inc() { x++; }
private: int x = 0;
};

class NewCounter : public Counter {
    int nr = 0;
public:
    void reset() override {   // override → kiểm tra tại biên dịch
        Counter::reset();     // gọi phiên bản lớp cha
        nr++;
    }
    int howMany() const { return nr; }
};

Ghi đè (Method Overriding)

Phương thức được chọn theo kiểu thực tế lúc chạy:


Counter*  p1 = new Counter();     // kiểu thực tế: Counter
Counter*  p2 = new NewCounter();  // kiểu thực tế: NewCounter

p1->reset();  // → Counter::reset()
p2->reset();  // → NewCounter::reset()  ← nhờ virtual
  • virtual — trình biên dịch trì hoãn việc chọn phương thức đến thời điểm chạy
  • override — kiểm tra tại biên dịch: xác nhận đang ghi đè đúng phương thức ảo

Cạm bẫy: Che mờ (Shadowing)

Bỏ virtual — lớp con che mờ phương thức, không ghi đè:


class Counter {
public:
    void reset() { x = 0; }   // KHÔNG có virtual
private: int x = 0;
};

class NewCounter : public Counter {
    int nr = 0;
public:
    void reset() {             // che mờ Counter::reset — KHÔNG phải ghi đè
        Counter::reset();
        nr++;
    }
};

Counter* p = new NewCounter();
p->reset();   // → Counter::reset()  ← NewCounter::reset() bị bỏ qua!
  • Kiểu tĩnh của con trỏ là Counter* → trình biên dịch gọi Counter::reset()
  • → Luôn dùng virtual + override khi muốn ghi đè

Kiểu con (Subtypes)

Định nghĩa
\(T\) là kiểu con của \(S\) (\(T <: S\)) nếu mọi thao tác của \(S\) đều có trong \(T\).
Giao diện của \(S\) ⊆ giao diện của \(T\).
  • Tương thích theo chức năng, không phải tên

Kiểu con — Ví dụ với Counter


class Counter {                       // S
public:
    virtual void reset() { x = 0; }
    int  get() { return x; }
    void inc() { x++; }
private: int x = 0;
};

class NewCounter : public Counter {   // T
public:
    void reset() override { Counter::reset(); nr++; }
    int  howMany() const  { return nr; }  // thêm thao tác mới
private: int nr = 0;
};
  • NewCounter có đủ thao tác của Counter (reset · get · inc) → NewCounter \(<:\) Counter
  • Dùng được ở mọi nơi cần Counter:

Counter* p = new NewCounter();  // OK — NewCounter là kiểu con của Counter
p->reset();  p->inc();  p->get();

Kiểu con ≠ Kế thừa

Kế thừa hỏi: "Lấy code từ đâu?"

  • NewCounter lấy get(), inc() từ Counter, không viết lại

Kiểu con hỏi: "Thay thế được không?"

  • NewCounter dùng được ở mọi nơi cần Counter 

Counter* p = new NewCounter(); // NewCounter vừa kế thừa, vừa là kiểu con của Counter
  • C++ public inheritance → tạo ra cả hai cùng lúc
  • Nhưng là hai câu hỏi độc lập — kế thừa không tự động là kiểu con

C++: Đa kế thừa

  • C++ cho phép đa kế thừa từ nhiều lớp cha
  • Tiềm ẩn mâu thuẫn nếu lớp cha có thành viên cùng tên

class A { public: void f() { std::cout << "A::f\n"; } };
class B { public: void f() { std::cout << "B::f\n"; } };
class C : public A, public B {
public:
    void g() {
        A::f();  // gọi f() của A
        B::f();  // gọi f() của B
    }
};
  • Nếu gọi f() từ đối tượng C: không biết gọi A::f() hay B::f() → lỗi biên dịch
  • → Giải pháp an toàn: kế thừa từ lớp all-pure-virtual (interface) — xem Phần 3

3.
Trừu tượng hóa
(Abstraction)

Trừu tượng hóa là gì?

Định nghĩa
Tách giao diện (what) khỏi cài đặt (how) — client biết phương thức làm gì, không cần biết làm thế nào.
  • Đóng gói ẩn dữ liệu — Trừu tượng hóa ẩn độ phức tạp cài đặt
  • Thay đổi cài đặt tự do — client không biết, không bị ảnh hưởng

Ví dụ: Counter


Counter c;
c.inc();  c.inc();
int v = c.get();  // client dùng inc/get/reset — không biết value lưu thế nào
  • Giao diện: reset() · get() · inc() — ổn định, không đổi
  • Cài đặt: int value — có thể thay bằng bất kỳ cách lưu trữ nào mà client không hay biết

Lớp trừu tượng (Abstract Class)

  • virtual f() = 0 — pure virtual: không có thân, lớp con bắt buộc cài
  • Lớp có ít nhất một pure virtual → lớp trừu tượng
    • Không thể thực thể hóa trực tiếp

class Shape {                              // lớp trừu tượng
public:
    virtual double area()      = 0;        // pure virtual
    virtual double perimeter() = 0;        // pure virtual
    virtual ~Shape() = default;
    void describe() {                      // phương thức thường — có thân
        std::cout << "area = " << area() << "\n";
    }
};
class Circle : public Shape {
    double r;
public:
    Circle(double r) : r(r) {}
    double area()      override { return 3.14159 * r * r; }
    double perimeter() override { return 2 * 3.14159 * r; }
};
// Shape s;          ← COMPILE ERROR: không thể thực thể hóa
Shape* s = new Circle(5);   // OK — con trỏ Shape, đối tượng Circle

Lợi ích: mở rộng không sửa client


// Code client — chỉ biết Shape*, không biết Circle hay Rectangle
void printAll(std::vector<Shape*>& shapes) {
    for (Shape* s : shapes)
        s->describe();    // area() đúng cho từng hình — nhờ virtual
}
Thêm hình mới — printAll không thay đổi:

class Rectangle : public Shape {
    double b, h;
public:
    Rectangle(double b, double h) : b(b), h(h) {}
    double area()      override { return b * h; }
    double perimeter() override { return 2 * (b + h); }
};

std::vector<Shape*> v = { new Circle(3), new Rectangle(4, 5) };
printAll(v);   // Circle::area(), Rectangle::area() — không sửa printAll

Interface

Interface = lớp trừu tượng mà toàn bộ thành viên đều pure virtual:

// Ví dụ: interface Drawable và Saveable            
class Drawable {
public:
    virtual void draw()        = 0;
    virtual void resize(int f) = 0;
    virtual ~Drawable() = default;
};
class Saveable {
public:
    virtual void save(const std::string& path) = 0;
    virtual ~Saveable() = default;
};
// Một lớp implement nhiều interface — "mix-in"
class Icon : public Drawable, public Saveable {
public:
    void draw()                        override { /* ... */ }
    void resize(int f)                 override { /* ... */ }
    void save(const std::string& path) override { /* ... */ }
};

4.
Đa hình
(Polymorphism)

Đa hình là gì?

Định nghĩa
Cùng một lời gọi phương thức → hành vi khác nhau tùy đối tượng nhận.
Phiên bản nào được chọn — quyết định tại thời điểm chạy, không phải biên dịch.
  • Điều phối động: chọn phương thức theo kiểu thực tế của đối tượng
  • Liên kết muộn: this liên kết với đối tượng thực tế khi chạy
  • Nền tảng: kiểu con + ghi đè (virtual) → điều phối tự động

Điều phối động

Điều phối động là cơ chế cốt lõi của OOP.

Nguyên lý: Chọn phiên bản phương thức theo kiểu thực tế (dynamic type) — không phải kiểu tĩnh của tham chiếu. 


Counter* c = new NewCounter(); // kiểu tĩnh: Counter*; kiểu thực tế: NewCounter
c->reset();  // → NewCounter::reset()  (nhờ virtual và override)

Counter* V[3] = { new Counter(), new NewCounter(), new NewCounter() };
for (Counter* v : V)
    v->reset(); // → mỗi đối tượng gọi đúng phiên bản reset() của mình

Liên kết muộn của this (Late Binding)

  • this / self: liên kết động với đối tượng nhận thông điệp
  • Phương thức lớp cha gọi được phương thức đã ghi đè ở lớp con

class Animal {
public:
    virtual void speak() { std::cout << "...\n"; }
    void introduce() {
        std::cout << "Hi, I say: ";
        this->speak();  // this liên kết động → gọi đúng phiên bản
    }
};
class Dog : public Animal {
public:
    void speak() override { std::cout << "Woof!\n"; }
};
Animal* a = new Dog();
a->introduce();  // in ra: "Hi, I say: Woof!"
  • Nếu speak() không có virtual → Animal::speak() được gọi, không phải Dog::speak()

Overloading vs. Điều phối động

Overloading — biên dịch

void print(int x);
void print(double x);
// Chọn theo kiểu tĩnh của đối số:
print(3);     // → print(int)
print(3.14);  // → print(double)
Dynamic dispatch — runtime

Counter* p = new NewCounter();
// Chọn theo kiểu thực tế của đối tượng:
p->reset(); // → NewCounter::reset()
  • Overloading: kiểu tĩnh của đối số → quyết định tại biên dịch
  • Virtual dispatch: kiểu thực tế của đối tượng → quyết định khi chạy

Tổng kết — Bốn trụ cột OOP

Đóng gói (Encapsulation)
Gom dữ liệu + phương thức; ẩn chi tiết nội bộ qua private.
→ Object · Class · Constructor
Kế thừa (Inheritance)
Tái dùng cài đặt (kế thừa) ≠ tương thích giao diện (kiểu con).
→ Subtype · Ghi đè · Đa kế thừa
Trừu tượng hóa (Abstraction)
Tách giao diện (what) khỏi cài đặt (how).
→ Lớp trừu tượng · Interface
Đa hình (Polymorphism)
Cùng lời gọi → hành vi khác nhau theo kiểu thực tế khi chạy.
→ Điều phối động · Late binding
