Giới thiệu:
Các mùi thiết kế

Một vài mùi thiết kế

"Code cũng có thể rối rắm như thế này — mùi thiết kế là dấu hiệu sớm để nhận ra."

Sự phình to (Bloaters)


class SuperUserManager { // Lớp quá lớn (Large Class)
public:
  // Danh sách tham số quá dài (Long Parameter List)
  void process(
      User& u, Database& db, EmailService& mail,
      Logger& log, int flag /*, ...*/) {
    // Hàm quá dài (Long Method): hàng ngàn dòng!
    // 1. Kiểm tra User (10 dòng)
    // 2. Format HTML Email (20 dòng)
    // 3. Lưu Database (100 dòng)
    // 4. Ghi Log (15 dòng)
    // ...
  }
};
  • Mù mờ (Opacity): Khó nhìn ra ý định cốt lõi của đoạn mã.
  • Dễ vỡ (Fragility): Sửa HTML có thể làm hỏng logic ở Database.

Cản trở thay đổi (Change Preventers)


class Person {
    std::string name;
    // ...
    std::string officeAreaCode;
    std::string officeNumber;
    std::string getTelephoneNumber() const {
        return std::string("(") + officeAreaCode + ") " + officeNumber;
    }
};

Nhận xét: Logic số điện thoại nằm trong Person — nên tách thành lớp TelephoneNumber riêng.

  • Tính cứng nhắc (Rigidity): Định dạng sđt thay đổi → bắt buộc sửa Person.
  • Tính bất động (Immobility): Muốn tái sử dụng logic sđt cho Employee, Supplier... → không tách ra được.
  • Sửa dây chuyền (Shotgun Surgery): Định dạng sđt đổi → phải sửa Person, Employee, Supplier, ... cùng lúc.

Sự liên kết quá mức (Couplers)


// Lấy khu vực giao hàng của đơn hàng
std::string region =
  order.customer()
       .address()
       .zipCode()
       .region();

Chuỗi thông điệp (Message Chains) — vi phạm Luật Demeter: "Chỉ nói chuyện với bạn bè trực tiếp."

  • Address bỏ ZipCode → phải sửa code gọi chuỗi này.
  • Dùng order ở dự án khác → kéo theo cả Customer, Address, ZipCode.

Câu lệnh switch (Switch Statement)


// Viết OOP nhưng tư duy Thủ tục!
void readData(DeviceType deviceType) {
    switch (deviceType) {
        case DeviceType::Keyboard:
            readKeyboard(); break;
        case DeviceType::PaperTape:
            readPaperTape(); break;
        // Thêm máy Scan?
        // → Mở file và thêm case!
    }
}
  • Thêm thiết bị mới → phải sửa khối switch
  • Lặp lại không cần thiết (Needless Repetition): khối switch bị copy-paste sang nhiều hàm

Mùi thiết kế → Nguyên lý SOLID

Mùi thiết kế Nguyên lý liên quan
Sự phình to — Lớp/hàm quá lớn S — Single Responsibility
Câu lệnh switch / Mã loại O — Open/Closed
Sửa dây chuyền (Shotgun Surgery) S — Single Responsibility
Phụ thuộc chặt vào chi tiết D — Dependency Inversion
Giao diện quá lớn (Fat Interface) I — Interface Segregation
Ép kiểu / kiểm tra kiểu thay vì đa hình L — Liskov Substitution

Đây là các nguyên lý thường liên quan, không phải ánh xạ một-một.

1.
Nguyên lý
Đơn Nhiệm

Khi một lớp biết làm tất cả

“Chỉ vì bạn có thể làm điều đó, không có nghĩa là bạn nên làm!”

Nguyên lý Đơn Nhiệm (SRP)

Định nghĩa (Robert C. Martin)
"A class should have only one reason to change."

→ Mỗi lớp chỉ phục vụ một actor — người/bộ phận duy nhất có quyền yêu cầu thay đổi lớp đó.

Trách nhiệm = Lý do thay đổi

  • Một trách nhiệm = một nhóm việc phục vụ cùng một actor.
  • Actor = người/bộ phận có quyền yêu cầu thay đổi code.
  • Một lớp, nhiều actor → nhiều lý do thay đổi → vi phạm SRP.

Vì sao cần SRP?

  • Thay đổi cô lập: thay đổi không lan dây chuyền.
  • Dễ kiểm thử: ít phụ thuộc, dễ cô lập.
  • Dễ đặt tên: tên lớp rõ, không cần chữ "AND".
  • Dễ tái sử dụng: lớp chuyên biệt dùng được ở nhiều nơi.
  • Cộng tác tốt hơn: các nhóm làm song song, ít xung đột.

Ví dụ: Employee

  • CFO yêu cầu đổi quy tắc tính lương → sửa calculatePay.
  • CTO yêu cầu đổi database → sửa store.
  • Hai actor, một lớp → hai lý do thay đổi.
  • Đổi lương không nên bắt ta sửa cùng lớp đang lo lưu trữ.

Khi nào nên tách?

  • NÊN TÁCH: Một phần đổi, phần kia không phải đổi theo.
    → Tách 2 lớp PayCalculator và EmployeeStore.
  • CHƯA CẦN TÁCH: Nếu hai phần vẫn luôn sửa cùng nhau.
    → Chưa vội tách.

Tự hỏi 2 điều:

  • Đổi cách tính lương, có phải sửa phần lưu không?
  • Đổi cách lưu, có phải sửa phần tính lương không?

Nếu không, nên tách.

Giải pháp: tách trách nhiệm

Trước: 1 lớp, 2 trách nhiệm


class Employee {
public:
    // TN 1: Nghiệp vụ
    Money calculatePay();


    // TN 2: Lưu trữ
    void store();
};

Sau: mỗi lớp 1 trách nhiệm


class PayCalculator {
public:
    Money calcPay(const Employee&);
};

class EmployeeStore {
public:
    void save(const Employee&);
};

Quy tắc nghiệp vụ thay đổi → chỉ sửa PayCalculator.
Đổi database → chỉ sửa EmployeeStore.

Dấu hiệu vi phạm SRP

SRP dễ hiểu nhưng khó làm đúng.

  • Lớp/hàm quá dài.
  • Tên cần chữ "AND".
  • Tên quá chung chung.
  • Sửa một chỗ, hỏng chỗ khác.
  • Một lớp vừa nghiệp vụ, kiểm tra dữ liệu, I/O.

Ví dụ: Lớp quá dài

  • Nhiều actor khác nhau cùng yêu cầu thay đổi.

class StudentService {
public:
    void validate();             // kiểm tra dữ liệu
    void calculateScholarship(); // học vụ
    void exportCsv();            // báo cáo
    void sendEmail();            // thông báo
    void writeLog();             // vận hành
};

💡 Tách: StudentValidator · ScholarshipCalculator · CsvExporter · EmailSender · Logger

Ví dụ: Tên có chữ "AND"

  • Chỉ nhìn tên đã thấy hai việc.

class SaveAndSendReport {
public:
    void save(const Report& r);
    void send(const Report& r);
};

💡 Tách: ReportRepository (lưu trữ) · ReportMailer (gửi email)

Ví dụ: Nghiệp vụ + kiểm tra + I/O

  • Ba loại việc → ba actor khác nhau.

class UserService {
public:
    bool validateEmail(std::string_view); // validation rules
    void registerUser(const User& u);    // business logic
    void writeLog(const User& u);        // logging / infra
};

💡 Tách: EmailValidator · UserRegistrar · AuditLogger

2.
Nguyên lý Mở Rộng / Hạn Chế

Mở rộng mà không sửa mã cũ

"Thêm tính năng bằng cách thêm code mới — không phải bằng cách sửa code cũ."

Định nghĩa OCP

Định nghĩa (Bertrand Meyer)
"Software entities should be open for extension, but closed for modification."
  • Mở (extension): Có thể thêm hành vi mới, không sửa code cũ.
  • Đóng (modification): Code đã ổn định thì không cần sửa.

So sánh các cách làm

Trên: sửa code cũ mỗi lần thêm
Dưới: thêm module mới mà không đụng code cũ.

Chìa khóa: trừu tượng hóa

"Làm sao để thay đổi những gì một mô-đun làm mà không thay đổi chính mã nguồn của nó?"

  • Tách phần cố định (logic điều phối) khỏi phần thay đổi (cách làm cụ thể).
  • Phần cố định phụ thuộc vào giao diện trừu tượng.
  • Phần thay đổi = lớp con cài đặt giao diện đó.

Ví dụ: chỉ đường

  • Cùng mục tiêu (đi đến đích) nhưng cách làm khác nhau.
  • Phần cố định: Navigator điều phối lộ trình.
  • Phần thay đổi: thuật toán từng phương tiện → tách ra.

Vấn đề của Navigator

Yêu cầu chính: lập lộ trình tự động.

  • Phiên bản 1: chỉ có đường bộ.
  • Thêm xe buýt → sửa Navigator.
  • Thêm xe đạp → lại sửa Navigator.

Kết quả: Mỗi thuật toán mới → Navigator lại phình to.

Mã vi phạm OCP


class Navigator {
public:
    enum class RouteType { Road, Walking, PublicTransit, Bicycle };

    void buildRoute(const std::string& start,
                    const std::string& end,
                    RouteType type) {
        if (type == RouteType::Road) {
            // ...
        } else if (type == RouteType::Walking) {
            // ...
        } else if (type == RouteType::PublicTransit) {
            // ...
        }
        // Muốn thêm xe đạp? → Mở file này ra và thêm else if!
        else if (type == RouteType::Bicycle) {
            // ...
        }
    }
};

Mỗi phương tiện mới = thêm if/else = mở Navigator ra sửa = vi phạm OCP.

Giải pháp: tách chiến lược

Gợi ý: Nhiều cách làm khác nhau → tách thành nhiều lớp.

  • Mỗi thuật toán là một chiến lược.
  • Navigator là bối cảnh.
  • Giữ tham chiếu đến một chiến lược.
  • Chỉ gọi qua RouteStrategy.

Mã giải pháp OCP


struct RouteStrategy {
    virtual ~RouteStrategy() = default;
    virtual void buildRoute(std::string start, std::string end) const = 0;
};

// Mỗi phương tiện = 1 lớp kế thừa RouteStrategy
class RoadStrategy    : public RouteStrategy { /* ... */ };
class BicycleStrategy : public RouteStrategy { /* ... */ };

class Navigator {
    RouteStrategy* strategy = nullptr;
public:
    void setStrategy(RouteStrategy* s) { strategy = s; }
    void buildRoute(std::string start, std::string end) const {
        strategy->buildRoute(start, end);
    }
};

Navigator đóng — thêm BicycleStrategy chỉ cần tạo lớp mới.

Mở gì? Đóng gì?

✓ Mở (thêm hành vi)
  • Thêm phương tiện mới: chỉ tạo lớp mới kế thừa RouteStrategy
  • Không đụng chạm mã cũ
✓ Đóng (không sửa mã cũ)
  • Navigator: không cần thêm if/else
  • Các chiến lược cũ: an toàn trước thay đổi

OCP không đóng tuyệt đối

  • Khách hàng đòi lộ trình kết hợp nhiều chặng.
  • Các chiến lược hiện tại chạy độc lập.
  • Muốn ghép lại → phải sửa Navigator hoặc giao diện.

Ghi nhớ: "Đóng" với thay đổi này chưa chắc "đóng" với thay đổi khác. Không có thiết kế nào đúng cho mọi tình huống.

Không trừu tượng hóa quá sớm

  • Chưa thấy thay đổi lặp lại → chưa cần tách.
  • Thấy cùng một kiểu sửa lặp lại từ lần thứ hai → bắt đầu trừu tượng hóa.
Bước 1
Chỉ có ô tô. Viết thẳng, chưa cần tách.
➔
Bước 2
Thêm xe đạp → thêm if/else. Chấp nhận được.
➔
Bước 3
Thêm xe buýt → lại sửa if/else. Thấy lặp lại!
➔
Bước 4
Tái cấu trúc → tách RouteStrategy.

Kích thích thay đổi sớm

Lưu ý: Trì hoãn quá lâu → code cứng lại → tái cấu trúc rất tốn công.

  • Viết kiểm thử sớm giúp lộ ra những chỗ thiết kế còn cứng nhắc.
  • Khó viết kiểm thử = tín hiệu thiết kế cần cải thiện.
  • Ví dụ: Navigator cần kết hợp nhiều chiến lược → tách thành MultiModalStrategy.

3.
Nguyên lý Thay Thế Liskov

Thay vào có an toàn không?

“Nếu nó trông như con vịt, kêu như con vịt, nhưng lại cần pin, thì có lẽ bạn đã thiết kế trừu tượng sai.”

Định nghĩa LSP

Kiểu con phải thay thế được kiểu cha.

Ở đâu dùng T, thay bằng S thì chương trình vẫn đúng.

  • Thay thế được = code gọi không cần biết đang dùng lớp cha hay lớp con.
  • Vi phạm → lỗi lúc chạy thay vì lúc biên dịch, code gọi bị bất ngờ.

Công cụ: Thiết kế theo hợp đồng

Biến giả định ngầm thành quy tắc rõ ràng.

Tiền điều kiện
Điều kiện để phương thức được gọi.
→ Lớp con không được đòi hỏi nhiều hơn.
Hậu điều kiện
Cam kết sau khi phương thức chạy xong.
→ Lớp con không được cam kết ít hơn.
Bất biến
Tính chất luôn đúng của đối tượng.
→ Lớp con phải duy trì bất biến của lớp cha.

Square có nên kế thừa Rectangle?

  • Toán học: hình vuông là một hình chữ nhật.
  • Rectangle cho phép đổi rộng và cao độc lập.
  • Square thì không.

Hậu điều kiện bị phá: setWidth(w) hứa width == w và height giữ nguyên. Square phá lời hứa đó → vi phạm LSP. 


class Square : public Rectangle {
public:
  void setWidth(double w) override {
    Rectangle::setWidth(w);
    Rectangle::setHeight(w); // giữ w = h
  }

  void setHeight(double h) override {
    Rectangle::setWidth(h);
    Rectangle::setHeight(h);
  }
};

void g(Rectangle& r) {
  r.setWidth(5);
  r.setHeight(4);
  // Mong đợi: width = 5, height = 4
  if (r.getWidth() * r.getHeight() != 20)
    throw std::runtime_error(“Contract broken”);
}

IS-A là về hành vi

  • Rectangle đứng riêng thì hợp lệ.
  • Square đứng riêng cũng hợp lệ.
  • Nhưng: Square phá giả định của code dùng Rectangle.
Quan hệ “là một” (IS-A) là về hành vi, không phải tên gọi.
Đúng về toán học ≠ đúng về thiết kế.

Cách sửa Rectangle/Square

  1. Không cho Square kế thừa Rectangle.
  2. Rút điểm chung lên Shape.
  3. Rectangle và Square kế thừa Shape riêng.

Kết quả: Giữ đúng hợp đồng — mỗi lớp đúng bản chất hành vi của nó.

Ví dụ: vi phạm tiền điều kiện


class TextDocument {
public:
    virtual void addText(std::string_view text) { // TĐK: chuỗi bất kỳ độ dài
        content += text;
    }
protected:
    std::string content;
};
class SmsDocument : public TextDocument {
public:
    void addText(std::string_view text) override { // TĐK MẠNH HƠN → vi phạm LSP!
        if (text.size() > 160) throw std::invalid_argument(“SMS too long”);
        content += text;
    }
};
Lớp cha: “Đưa tôi chuỗi dài bao nhiêu cũng được”
Lớp con: “Tôi chỉ nhận tối đa 160 ký tự”
→ Tiền điều kiện mạnh hơn → vi phạm LSP!

🤔 Thảo luận

Câu hỏi: Đoạn code này có vi phạm LSP không?


class Bird {
public:
    virtual void fly() { /* bay... */ }
};

class Penguin : public Bird {
public:
    void fly() override {
        throw std::logic_error(“Chim cánh cụt không bay được!”);
    }
};

💡 Gợi ý: hàm nhận Bird& gọi fly() có hoạt động đúng với Penguin không?

4.
Nguyên lý Phân Tách Giao Diện

Đừng bắt lớp làm việc thừa

“Mã sử dụng không nên phụ thuộc vào những giao diện mà nó không dùng!”

Định nghĩa ISP

Đừng ép mã sử dụng phụ thuộc vào những gì nó không dùng.
  • Cám dỗ: Gom hết vào một giao diện lớn.
  • Hậu quả: Ép các lớp cài phương thức không liên quan.

Vấn đề: Door và TimerClient

  • Door: khóa, mở khóa, kiểm tra mở.
  • TimedDoor: thêm báo động khi mở quá lâu.
Hậu quả
  • NormalDoor phải cài timeout() vô nghĩa.
  • Door bị gắn chặt với bộ định thời.
  • Mã sử dụng phụ thuộc nhiều hơn mức cần.

struct TimerClient {
    virtual ~TimerClient() = default;
    virtual void timeout() = 0;
};

// SAI: Ép Door kế thừa TimerClient
struct Door : TimerClient {
    virtual ~Door() = default;
    virtual void lock() = 0;
    virtual void unlock() = 0;
    virtual bool isOpen() const = 0;
};

class NormalDoor : public Door {
public:
    void lock() override { /*...*/ }
    void unlock() override { /*...*/ }
    bool isOpen() const override { return false; }

    void timeout() override {
      // Cài đặt thừa
    }
};

Giải pháp: tách vai trò

NormalDoor không còn bị ép cài timeout(). Mỗi lớp chỉ phụ thuộc giao diện nó thực sự dùng. 


// Tách thành 2 giao diện nhỏ, độc lập
struct Door {
    virtual void lock() = 0;
    virtual void unlock() = 0;
};
struct TimerClient { virtual void timeout() = 0; };

// Mỗi lớp chỉ kế thừa giao diện nó cần
class TimedDoor : public Door, public TimerClient {
public:
    void lock() override { /* ... */ }
    void unlock() override { /* ... */ }
    void timeout() override { /* bao dong */ }
};
class NormalDoor : public Door {
public:
    void lock() override { /* ... */ }
    void unlock() override { /* ... */ }
};

🤔 Thảo luận

Câu hỏi: Giao diện này có vi phạm ISP không? Tách như thế nào?


struct Animal {
    virtual void eat() = 0;
    virtual void fly() = 0;
    virtual void swim() = 0;
    virtual void run() = 0;
};

💡 Gợi ý: Dog có cần implement fly()? Eagle có cần swim()?

Ví dụ: ATM

  • Giao diện người dùng (UI) chung quá lớn.
  • Thêm giao dịch mới → thêm phương thức mới → nhiều lớp phải biên dịch lại.
  • Mỗi giao dịch chỉ dùng một phần nhỏ của giao diện đó.

Giải pháp: tách giao diện

  • Tách thành DepositUI, WithdrawUI, TransferUI, ...
  • Mỗi giao dịch chỉ phụ thuộc phần nó cần.

Lợi ích: Thêm IElectricBillUI → chỉ ảnh hưởng phạm vi nhỏ.

5.
Nguyên lý Nghịch Đảo Phụ Thuộc

Ai nên phụ thuộc vào ai?

“Bạn có hàn trực tiếp một chiếc bóng đèn vào dây điện trong tường không?”

Định nghĩa DIP

  • Mô-đun cấp cao và cấp thấp cùng phụ thuộc vào sự trừu tượng.
  • Chi tiết phải phụ thuộc vào sự trừu tượng, không phải ngược lại.

Tại sao gọi là "Đảo ngược"?

  • Phương pháp truyền thống: Mô-đun cấp cao phụ thuộc vào cấp thấp → chính sách phụ thuộc vào chi tiết.
  • Hậu quả: Chi tiết đổi → chính sách bị ảnh hưởng.

Phụ thuộc kiểu cũ

  • Policy → Mechanism → Utility.

Vấn đề: Nâng cấp SQL driver ở tầng dưới cũng có thể kéo tầng trên biên dịch lại.

  • Logic nghiệp vụ bị khóa chặt vào thư viện tầng dưới.

Khi phụ thuộc đảo chiều

  • Tầng chính sách không biết tầng cơ chế là ai.
  • Chỉ làm việc qua giao diện do Policy định nghĩa.
  • Đổi tầng dưới → tầng chính sách vẫn ổn.

Kết quả: Đổi tầng cơ chế (ví dụ thay SQL bằng NoSQL) mà tầng chính sách không cần sửa.

Ai sở hữu giao diện?

Điểm quan trọng: Ai sở hữu giao diện thì người đó làm chủ.

  • Cách cũ: Tầng dưới đưa giao diện, tầng trên phải theo.
  • Cách mới: Tầng trên đưa yêu cầu, tầng dưới phải theo.

Nguyên tắc Hollywood: "Đừng gọi chúng tôi, chúng tôi sẽ gọi bạn"

  • Ưu điểm: Đổi cơ chế bên dưới mà không phải sửa tầng chính sách.

Tầng trên định nghĩa giao diện

Cách 1 — Tầng dưới đặt giao diện ✗

// Database đặt giao diện
struct ISqlDatabase {
    virtual ~ISqlDatabase() = default;
    virtual void execute(const std::string& sql) = 0;
};

// OrderService dính SQL
class OrderService {
    ISqlDatabase& db;
public:
    explicit OrderService(ISqlDatabase& db) : db(db) {}

    void processOrder(const Order& order) {
        // Trộn nghiệp vụ với SQL
        db.execute("INSERT INTO Orders...");
    }
};
Cách 2 — Tầng trên đặt giao diện ✓

// OrderService đặt nhu cầu của mình
struct IOrderRepository {
    virtual ~IOrderRepository() = default;
    virtual void save(const Order& order) = 0;
};

class OrderService {
    IOrderRepository& repo;
public:
    explicit OrderService(IOrderRepository& repo) : repo(repo) {}

    void processOrder(const Order& order) {
        repo.save(order); // Không biết SQL
    }
};

class SqlOrderRepository : public IOrderRepository {
public:
    void save(const Order& order) override {
        // "INSERT INTO Orders..."
    }
};

Vì sao cách 2 tốt hơn?

  • Đổi SQL → chỉ sửa SqlOrderRepository.
  • OrderService và IOrderRepository giữ nguyên.
  • Tầng dưới đi theo giao diện của tầng trên.

Chốt: Đừng thiết kế theo công cụ. Hãy thiết kế theo nhu cầu nghiệp vụ.

Tổng kết

Các nguyên lý SOLID

S — Single Responsibility (SRP)
Một lớp chỉ làm một việc chính.
→ Phụ thuộc thấp, gắn kết cao
O — Open / Closed (OCP)
Mở để mở rộng, đóng để sửa đổi.
→ Trừu tượng hóa · Tách chiến lược
L — Liskov Substitution (LSP)
Kiểu con phải thay thế được kiểu cha.
→ IS-A về hành vi · Thiết kế theo hợp đồng
I — Interface Segregation (ISP)
Mã sử dụng không dùng phương thức nào thì đừng ép cài đặt.
→ Giao diện nhỏ · tập trung
D — Dependency Inversion (DIP)
Phụ thuộc vào sự trừu tượng, không phụ thuộc vào cài đặt cụ thể. Lớp cấp cao sở hữu giao diện.
→ Nguyên tắc Hollywood · Đảo ngược quyền sở hữu
