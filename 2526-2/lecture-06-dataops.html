Gợi ý từ học liệu
Dữ liệu là nền tảng của hệ thống ML/AI. Bài toán “data” không chỉ là lưu trữ, mà là quy trình, chất lượng, quan sát được, và tái lập.

1.
DataOps & kiến trúc dữ liệu cho dự án AI

Vì sao “DataOps” quan trọng trong AI?

  • Hệ thống AI không chỉ là mô hình; phần “data stack” quyết định khả năng vận hành.
  • Dữ liệu đến từ nhiều nguồn, nhiều dạng, nhiều nhịp (batch/stream).
  • AI thất bại âm thầm: dữ liệu sai/thiếu/lệch → mô hình vẫn chạy nhưng kết quả xuống cấp.
  • Do đó cần một cách làm: tự động hoá + kiểm soát chất lượng + quan sát cho dữ liệu.

Định nghĩa làm việc: DataOps

Định nghĩa
DataOps là cách tiếp cận kết hợp Agile + DevOps + tự động hoá kiểm soát chất lượng để rút ngắn vòng đời thay đổi dữ liệu (data changes) và tăng độ tin cậy của pipeline dữ liệu.

Trọng tâm: văn hoá & quy trình + tự động hoá, không chỉ là “cài thêm công cụ”.

Khái niệm Đối tượng chính Đầu ra

Kiến trúc dữ liệu điển hình cho AI

Mục tiêu: tách “storage” khỏi “compute”, và tách “serving” khỏi “analytics” khi cần.

DataOps checklist (mức tối thiểu)

  • Version control: pipeline code, schema, config, docs.
  • CI/CD cho data: chạy test dữ liệu khi PR thay đổi pipeline/transform.
  • Data quality: schema checks, null/outlier, constraints, freshness.
  • Observability: latency, volume, error rate, drift, SLA.
  • Lineage: biết “dữ liệu này đến từ đâu, biến đổi thế nào”.
  • Security: phân quyền, audit log, PII/PHI, retention.

2.
Storage & Database cho AI

Dữ liệu đến từ đâu?

  • User input: text/image/file upload → dễ “malformatted”, cần validate mạnh.
  • System-generated: logs, metrics, predictions → quan trọng cho debug/monitor.
  • Internal DB: inventory, users, payments, CRM…
  • Third-party: dữ liệu mua/đối tác → cần kiểm tra license & chất lượng.

Nguồn đa dạng → format + schema + access pattern đa dạng.

Format dữ liệu: text vs binary

  • Text (CSV/JSON): dễ đọc, tiện debug, nhưng tốn dung lượng và thường chậm.
  • Binary (Parquet/Avro/Protobuf): nén tốt, I/O tốt hơn, nhưng cần tooling.
Format Text/Binary Điển hình
CSV Text trao đổi đơn giản
JSON Text API payload, logs
Parquet Binary data lake/warehouse
Protobuf / TFRecord Binary ML pipelines

Row-major vs Column-major

  • Row-major (CSV): ghi/append theo “bản ghi” nhanh, tốt cho transactional-like.
  • Column-major (Parquet): đọc theo “cột/feature” nhanh, tốt cho analytics/ML.
  • Trong ML: dữ liệu thường là “nhiều features” → columnar rất hợp.

Data model: relational vs document vs graph

  • Relational: schema rõ, SQL mạnh; join có thể đắt ở quy mô lớn.
  • Document: linh hoạt schema, locality tốt; join khó hơn.
  • Graph: truy vấn quan hệ nhiều-bước hiệu quả (social, knowledge graph).

Chọn data model đúng → code đơn giản hơn, hiệu năng tốt hơn.

Workload: OLTP vs OLAP (tư duy)

  • OLTP: giao dịch online, latency thấp, availability cao (user-facing).
  • OLAP: phân tích, aggregation, scan theo cột, truy vấn “nặng”.
  • Trong AI: thường cần cả hai:
    • OLTP để vận hành sản phẩm
    • OLAP để training/analytics/BI

Tách storage và compute

  • Một xu hướng: decouple nơi lưu dữ liệu (storage) khỏi engine xử lý (compute).
  • Lợi ích: dùng cùng dữ liệu cho nhiều kiểu workload; scale compute độc lập.
  • Ví dụ: warehouse cloud, lakehouse (storage chung + nhiều engine SQL/stream).

Data lake, warehouse, lakehouse

Khái niệm Tập trung Phù hợp
Data lake raw/unstructured lưu rẻ, ingest nhanh, linh hoạt
Data warehouse structured/curated BI/SQL, dữ liệu “sạch”, governance
Lakehouse kết hợp muốn vừa linh hoạt vừa quản trị

Kho lưu trữ cho ML: “lưu” để “lấy lại”

  • Lưu trữ chỉ có ý nghĩa nếu truy xuất lại được đúng thứ mình cần.
  • Cần quan tâm:
    • schema (đúng nghĩa dữ liệu)
    • index (tốc độ)
    • partition (chi phí scan)
    • retention (lưu bao lâu)
    • metadata (ai dùng? dùng cho gì?)

Dataflow modes (trong hệ phân tán)

  • Qua database: đơn giản nhưng có thể chậm cho latency nghiêm ngặt.
  • Qua service requests (REST/RPC): phù hợp microservices.
  • Qua real-time transport (Kafka/Kinesis): stream-first, decouple producers/consumers.

Chọn mode theo: latency, coupling, throughput, correctness (ordering/exactly-once?).

  2. Trong dự án AI, thành phần nào thường là OLTP? thành phần nào thường là OLAP?
  3. Khi nào “streaming” là bắt buộc (không chỉ là “nice to have”)?

3.
Vector DB & retrieval theo embedding

Tại sao vector DB xuất hiện?

  • Nhiều bài toán AI cần tìm kiếm theo độ giống (similarity) thay vì exact match.
  • Embedding biến đối tượng (text/image/audio/code) thành vector \( \mathbb{R}^d \).
  • Truy vấn: “tìm top-k vector gần nhất” theo cosine/L2/dot product.
  • Khi dữ liệu lớn: brute force quá chậm → cần index ANN.

Vector DB lưu gì?

  • Vector: embedding (dimension d)
  • Metadata: id, timestamp, source, tags, ACL, namespace/tenant…
  • Index: cấu trúc để truy vấn ANN nhanh (HNSW, IVF, PQ...)

ANN index: HNSW

  • HNSW = graph đa tầng; tìm kiếm “đi từ thưa đến dày” để nhanh.
  • Ưu: recall tốt, latency thấp; nhược: tốn RAM, build/indexing có chi phí.
  • Thông số hay gặp:
    • M: số liên kết mỗi node (tăng M → tăng recall, tăng RAM)
    • efConstruction: độ sâu khi xây index
    • ef (query-time): độ sâu khi truy vấn

ANN index: IVF, PQ (ý tưởng)

  • IVF: phân cụm (k-means) → chỉ tìm trong vài cụm gần nhất.
  • PQ: lượng tử hoá (compress) vector để tiết kiệm bộ nhớ.
  • Đánh đổi: tốc độ ↔ recall ↔ chi phí build ↔ chi phí RAM.

Vector DB vs “tự làm” (FAISS + file)

  • Vector DB thường thêm:
    • metadata filtering (namespace, time, tags)
    • persistence + replication
    • multi-tenant + ACL
    • monitoring + backups
  • Nếu chỉ demo nhỏ: có thể FAISS + SQLite; nếu sản phẩm: cần vận hành & SLA.

Điểm hay “vỡ” trong thực tế

  • Embedding drift: model embed đổi phiên bản → vector cũ không còn “cùng không gian”.
  • Chunking kém: retrieve sai ngữ cảnh dù index tốt.
  • Metadata không chuẩn: filter không được → leakage / truy hồi nhầm tenant.
  • Freshness: dữ liệu mới chưa index kịp → trả lời lạc hậu.

Thiết kế lifecycle cho embeddings

  • Quy ước embedding_version (như schema version).
  • Chiến lược re-embed:
    • offline re-embed toàn bộ (batch)
    • dual index (v1 + v2) rồi cutover
    • lazy re-embed theo truy cập (khó)
  • Log: truy vấn → top-k → feedback để đánh giá.

  3. Nếu hệ thống phải lọc theo quyền truy cập (ACL), metadata nên thiết kế thế nào?

4.
ETL/ELT, pipeline và orchestration

ETL là gì?

  • Extract: lấy dữ liệu từ nguồn; validate; loại bỏ dữ liệu lỗi.
  • Transform: làm sạch, join, chuẩn hoá, dedup, aggregate, derive feature…
  • Load: nạp vào đích (file/DB/warehouse), theo lịch/trigger.

ETL là “khung” nền của nhiều data layer trong tổ chức.

ELT là gì? (đảo thứ tự)

  • ELT: Extract → Load raw trước → Transform sau.
  • Ưu: ingest nhanh, linh hoạt; nhược: “raw lake” quá lớn → tìm đúng dữ liệu tốn chi phí.
  • Thực tế: thường kết hợp + lakehouse để cân bằng.

Pipeline trong AI khác pipeline BI thế nào?

  • BI: transform → metric/report.
  • AI: transform → labeling → feature/embedding → train/eval → deploy → feedback.
  • AI đòi hỏi:
    • reproducibility
    • training-serving consistency
    • monitoring drift

Một số data pipeline mẫu (theo loại dữ liệu)

  • Mỗi loại dữ liệu có đặc thù lưu trữ + biến đổi + quality checks khác nhau.
  • Chọn “pipeline mẫu” giúp:
    • định hình nhanh các bước cần có
    • đặt đúng chỗ cho kiểm thử & versioning
    • tách phần chung (ingest/store/monitor) và phần riêng (transform theo modality)

Pipeline thu thập dữ liệu (collection)

  • Nguồn phổ biến:
    • Public datasets: Kaggle, UCI…
    • API: Twitter, Maps…
    • Web scraping: BeautifulSoup, Scrapy…
  • Guardrails tối thiểu:
    • validate định dạng + loại bỏ nhiễu
    • đảm bảo dữ liệu phù hợp bài toán + tuân thủ license

Pipeline dữ liệu hình ảnh (ví dụ)

  • Thách thức: file lớn, cần annotation, bias/cân bằng lớp.
  • Transform điển hình:
    • resize về kích thước chuẩn
    • chuẩn hoá pixel (ví dụ /255)
    • augmentation: crop/flip/rotate/brightness
  • Load: object storage (S3/MinIO) + metadata DB + label store.

Pipeline dữ liệu văn bản (ví dụ)

  • Thách thức: encoding, dữ liệu không cấu trúc, đa ngôn ngữ.
  • Transform điển hình:
    • normalization (lowercase, chuẩn hoá unicode)
    • tokenization; stopwords; stemming/lemmatization (tuỳ ngôn ngữ)
    • vectorization: TF-IDF hoặc embeddings
  • Load: lake/warehouse (Parquet) + index (search/vector DB) nếu cần truy hồi.

Pipeline dữ liệu âm thanh (ví dụ)

  • Thách thức: nhiễu, nhiều định dạng, giọng/phương ngữ.
  • Transform điển hình:
    • chuẩn hoá định dạng (WAV) + resampling
    • noise reduction
    • trích đặc trưng: MFCC, spectrogram
  • Load: object storage + metadata (duration, sample rate) + transcript/labels nếu có.

Pipeline dữ liệu chuỗi thời gian (ví dụ)

  • Thách thức: missing values, sampling không đều, trend/seasonality.
  • Transform điển hình:
    • imputation/interpolation hoặc forward fill
    • resampling về tần suất cố định
    • feature time-based: lag features, rolling statistics
  • Load: time-partitioned tables + retention policy rõ ràng.

Pipeline dữ liệu có cấu trúc (tabular)

  • Transform điển hình:
    • xử lý missing (mean/median/mode hoặc model-based)
    • encoding categorical (one-hot/label)
    • scaling numeric (min-max, z-score)
    • outlier detection/handling
  • Quality tests: schema/constraints + kiểm tra phân phối theo thời gian.

Ví dụ ETL + DVC cho dữ liệu hình ảnh

  • Extract: thu thập ảnh (ví dụ Flickr API).
  • Transform: resize 256×256, chuẩn hoá pixel, augmentation.
  • Load: lưu ảnh đã xử lý vào object storage (ví dụ S3).
  • DVC: theo dõi version dataset + chia sẻ remote:
    • dvc init, dvc add data/
    • git commit (metadata), dvc push (data)

Điểm mấu chốt: “code trong Git” + “data ngoài Git” nhưng được version hoá/trace được.

Thiết kế pipeline “dạy được” (nguyên tắc)

  • Rõ input/output cho từng stage.
  • Idempotent nếu có thể (chạy lại không phá).
  • Deterministic cho transform (cùng input → cùng output).
  • Backfill được (lấp dữ liệu quá khứ).
  • Observable: log/metric cho từng stage.

Batch vs Stream (tư duy xử lý)

Batch Stream
Độ trễ phút/giờ/ngày ms/giây
Đơn vị file/partition event
Độ phức tạp thấp hơn cao hơn (ordering, late events)
Use case AI training, backfill feature online, realtime monitoring

Orchestration: cần khi nào?

  • Khi pipeline có nhiều bước + phụ thuộc + retry + schedule.
  • Cần:
    • dependency graph (DAG)
    • retries + backoff
    • alerting khi fail/SLA
    • quản lý secrets/credentials

DataOps “CI/CD cho dữ liệu”

  • CI cho pipeline: mỗi PR chạy:
    • lint + unit tests transform
    • schema validation
    • data quality tests trên sample
  • CD: deploy theo môi trường (dev/staging/prod) + kiểm soát rollback.
  • Khác code: dữ liệu “stateful” và thường không rollback đơn giản.

Thiết kế để “giảm chi phí”

  • Partitioning: theo ngày/tenant/region để tránh full scan.
  • Incremental: chỉ xử lý phần mới thay vì “recompute all”.
  • Materialize: lưu kết quả transform đắt.
  • Cache: cho feature online / retrieval.

5.
Feature, Embedding & Feature Store

Feature là gì trong hệ thống AI?

  • Feature = thông tin đầu vào cho mô hình sau khi “biến đổi đúng dạng”.
  • Feature engineering thường đem lại boost lớn khi đã có model “chạy được”.
  • Thực tế: feature không chỉ từ text/image; còn từ user, hành vi, context.

Embedding: learned features (tóm tắt)

  • Embedding là “đặc trưng học được” từ dữ liệu (ví dụ word embeddings).
  • Ưu: nén thông tin, hỗ trợ similarity; nhược: khó giải thích, drift theo model.
  • Trong sản phẩm AI hiện đại: embedding là “tài sản” cần quản trị như dữ liệu.

Training-serving skew (vấn đề cổ điển)

Cảnh báo
Nếu cách tạo feature khi train khác cách tạo feature khi infer → mô hình “đúng trong offline” nhưng “sai trong production”.
  • Ví dụ: scaling khác, encoding khác, join bị lệch thời gian, leakage.

Feature store: 3 bài toán lõi

  • Feature management: chia sẻ/khám phá/permission.
  • Feature computation: tính feature và lưu lại để tái sử dụng.
  • Feature consistency: dùng chung logic batch + stream để đồng nhất train & serve.

Feature store là khái niệm “loaded”: vendor khác nhau có thể làm khác nhau.

Offline store vs Online store

Offline Online
Mục tiêu training, backfill low-latency inference
Dữ liệu historical, lớn mới, nhỏ hơn
Công nghệ lake/warehouse KV store / cache / DB tối ưu đọc
SLA phút/giờ ms

Feature catalog vs Feature store

  • Feature catalog: quản lý metadata + khám phá.
  • Feature store: có thể bao gồm catalog + compute + store + consistency.
  • Công cụ “catalog/metadata” thường gặp: DataHub, Amundsen…

Design: feature theo thời gian (time travel)

  • Nhiều feature phụ thuộc thời điểm (ví dụ “7 ngày gần nhất”).
  • Train cần “đúng thời điểm trong quá khứ” để tránh leakage.
  • Online cần “cập nhật đủ nhanh” để phục vụ infer.

Từ góc DataOps: define SLA về freshness và kiểm tra tự động.

Embedding store & vector DB: quan hệ

  • Embedding cũng là “feature”, nhưng truy vấn theo similarity.
  • Vector DB giống “online store” chuyên cho embedding retrieval.
  • Offline: lưu embedding theo phiên bản để train/eval; Online: index để truy hồi.

Kịch bản 1: Search/RecSys

  • Offline:
    • tính user/item features
    • tính embeddings
    • train ranking model
  • Online:
    • lấy feature realtime + vector retrieval
    • rank + serve

Kịch bản 2: Fraud/Anomaly

  • Feature thường “cửa sổ thời gian” (rolling windows).
  • Stream processing quan trọng để cập nhật hành vi mới.
  • Cần thiết kế “late events” + exactly-once/at-least-once hợp lý.

  2. Training-serving skew thường đến từ đâu?

6.
Data quality, observability & lineage

Data quality: không phải “1 lần là xong”

  • Dữ liệu thay đổi theo thời gian: schema drift, distribution drift, missing values…
  • Chất lượng cần được kiểm thử tự động như code.
  • Hệ thống AI còn cần kiểm: feature distribution, label quality, bias.

Các lớp kiểm thử dữ liệu (gợi ý)

  • Schema tests: kiểu dữ liệu, cột bắt buộc, enum.
  • Constraint tests: range, uniqueness, referential integrity.
  • Statistical tests: phân phối, drift, outliers.
  • Freshness & volume: dữ liệu có đến đúng giờ và đủ lượng không?
  • Business rules: logic miền nghiệp vụ.

Observability cho data pipeline

  • Mục tiêu: trả lời nhanh “cái gì hỏng?” và “ảnh hưởng đến ai?”.
  • Metric điển hình:
    • latency từng stage
    • error rate + retry count
    • freshness lag
    • row count, null rate

Metadata & catalog

  • Metadata trả lời: dataset này là gì? ai sở hữu? dùng cho gì? SLA?
  • Catalog giúp:
    • khám phá dữ liệu (search)
    • quản trị quyền truy cập
    • liên kết lineage và quality

Lineage: dataset → job → run

Ý tưởng
Lineage mô tả “dữ liệu nào” được tạo bởi “job nào” ở “lần chạy nào”, và phụ thuộc input nào.
  • Giúp debug: biết bảng A sai do job X hay do upstream Y.
  • Giúp governance: đánh giá tác động khi thay đổi schema/pipeline.

OpenLineage & Marquez (ví dụ)

  • OpenLineage: chuẩn mở mô tả lineage metadata (dataset/job/run + facets).
  • Marquez: triển khai tham chiếu (metadata service + UI + API).
  • Thực tế: lineage nên “cắm” vào orchestration và compute (Airflow/Spark/dbt…).

Data quality ↔ Lineage (kết hợp)

  • Data quality fail + lineage → tự động xác định “blast radius”.
  • Ví dụ:
    • bảng `features_daily` drift → model training A/B bị ảnh hưởng
    • stream consumer delay → feature online stale → latency tăng

Checklist vận hành: incident “data”

  • Detect: alert drift/freshness/volume.
  • Diagnose: lineage + logs để tìm upstream root cause.
  • Mitigate: rollback data? pause downstream? serve fallback?
  • Recover: backfill + re-train/re-index.
  • Prevent: thêm test, thêm guardrail, chỉnh SLA.

Câu hỏi nhanh

  1. “Freshness” có quan trọng cho mọi feature không? feature nào cần freshness cao?
  2. Lineage ở mức file/bảng đã đủ chưa? khi nào cần row-level provenance?
  3. Nếu drift xảy ra, bạn muốn cảnh báo theo thống kê nào?

7.
Data version control, provenance & reproducibility

Vì sao cần versioning/provenance?

  • Model thay đổi liên tục; dữ liệu thay đổi liên tục.
  • Khi có sự cố (bias, lỗi, khiếu nại): cần truy vết “model nào” dùng “data nào” do “pipeline nào” tạo.
  • Nếu chỉ “overwrite model” trong production → sớm muộn sẽ mất dấu và không audit/debug được.

Ba khái niệm

  • Versioning: phân biệt các phiên bản (revisions + variants).
  • Provenance/Lineage: theo dõi nguồn gốc & biến đổi dữ liệu.
  • Reproducibility: tái tạo lại kết quả đủ mức cần thiết (debug/audit).

Versioning cái gì trong dự án AI?

  • Data: raw + curated + labels + embeddings.
  • Pipeline code: ETL/feature/embedding/training.
  • Model artifacts: weights, configs, metrics, tags.
  • Infra config: container, IaC, secrets policy (không commit secrets).
  • Inference API: version endpoint để client kiểm soát cập nhật.

Versioning dataset lớn: 5 chiến lược

  • Copy whole: dễ truy xuất, tốn storage.
  • Store deltas: tiết kiệm nếu thay đổi nhỏ, restore có thể đắt.
  • Append-only offsets: hợp logs/streams.
  • Per-record version: version theo key (phù hợp KV/records).
  • Recreate via pipeline: view/derived data (cần determinism + version input + code).

Tools hay gặp

  • DVC: version data + pipeline stages + metadata trong Git.
  • lakeFS: “git-like” cho data lake.
  • Delta Lake/Iceberg/Hudi: time travel + table versioning (lakehouse).
  • MLflow/W&B/Neptune: experiment tracking + model registry.

Provenance trong pipeline: 4 cách

  • Manual docs: nhanh nhưng dễ lỗi thời.
  • Custom metadata logs: linh hoạt nhưng cần kỷ luật.
  • Framework-enforced (DVC/Flyte/Pachyderm): tự track nếu “đi đúng đường”.
  • Integrated platform: track tự động nếu mọi thứ chạy trên platform.

Reproducibility vs Replicability

  • Replicability (exact): chạy lại đúng y chang (bit-for-bit) — hữu ích cho debug.
  • Reproducibility (similar): lặp lại thí nghiệm, kết quả tương đương — quan trọng cho khoa học.
  • Trong production: thường ưu tiên “đủ để debug/audit” hơn là tuyệt đối bit-for-bit.

Nondeterminism: đến từ đâu?

  • Training (SGD, random init, shuffling) thường không deterministic.
  • Thiếu kỷ luật versioning:
    • data thay đổi nhưng không có snapshot
    • library version trôi
    • feature code thay đổi nhưng output không ghi nhận
  • Giải pháp: pin deps, log versions, snapshot data quan trọng, track seeds khi cần.

Câu hỏi nhanh

  1. Nếu model tạo quyết định gây tranh cãi, bạn cần log tối thiểu những gì?
  2. Dataset nào nên snapshot “copy whole”, dataset nào nên append-only?
  3. Nếu team không có time làm DVC, bạn sẽ bắt đầu tối thiểu từ đâu?

Tổng kết

  • DataOps: kết hợp quy trình + tự động hoá để pipeline dữ liệu tin cậy.
  • Storage/DB: hiểu format, model, workload, và dataflow modes.
  • Vector DB: phục vụ retrieval theo embedding; cần quản trị lifecycle embedding.
  • ETL/ELT: nền tảng của data layer; pipeline cần deterministic + observable.
  • Feature store: quản lý feature, compute, consistency (giảm training-serving skew).
  • Quality/Lineage/Versioning: để debug, audit, và vận hành “đỡ đau”.
