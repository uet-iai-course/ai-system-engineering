Mục tiêu buổi học

  • Hiểu MLOps như một hệ thống để cập nhật model nhanh và an toàn.

0.
Động lực MLOps &
Vòng đời

Câu hỏi: "MLOps là gì?"

Định nghĩa làm việc
MLOps là tập thực hành để tự động hoá và vận hành vòng đời ML: data → train → eval → release → deploy → monitor → update.

Vấn đề #1: ML có thể “fail silently”

  • Không crash, không exception — nhưng dự đoán sai.
  • Người dùng khó phát hiện nếu không có đo lường.
  • → Monitoring và test-in-production là bắt buộc.

Vấn đề #2: Dữ liệu không đứng yên

  • Training data và production data khác nhau theo thời gian.
  • Model “rot”: chất lượng giảm dần/sai theo phân khúc.
  • → Hệ thống phải hỗ trợ update liên tục.

Vấn đề #3: Nhiều thành phần → khó debug

  • Code + data + labels + features + configs + infra.
  • Một lỗi nhỏ ở pipeline tạo model “hợp lệ” nhưng sai.
  • → Cần lineage: model đến từ run/code/data nào.

DevOps vs MLOps

Khía cạnh DevOps MLOps
Artifact binary model + code + data snapshot
Test deterministic statistical + tolerances
Thay đổi code code + data + model
Quan sát logs/metrics + drift + model quality

Câu hỏi: "Production là gì?"

Một phổ (spectrum)
Từ "notebook cho báo cáo" → đến "dịch vụ realtime 99.9% uptime".
  • MLOps cần thiết khi có SLA (Service Level Agreement – cam kết mức dịch vụ) / latency, update thường xuyên, nhiều model.
  • Nếu đơn giản: vẫn cần versioning + tracking tối thiểu.

Ba thói quen “tối thiểu”

  • Log mọi run: config + metrics + artifacts.
  • Version hóa: code/data/model/environment.
  • Đặt gates: không promote nếu chưa qua checks tối thiểu.

1.
Training &
Experiment Tracking

Câu hỏi: "Vì sao tracking?"

Mục tiêu
Khi có sự cố / cần cải tiến, trả lời nhanh: model này train bằng data nào, config nào, code nào.

Những thứ cần log cho mỗi run

  • Params: lr, batch size, seed…
  • Metrics: loss/accuracy/AUC (Area Under ROC Curve – diện tích dưới đường cong ROC), latency, cost…
  • Artifacts: checkpoint, plots, reports…
  • Environment: Python/lib versions, CUDA…

Quản lý cấu hình: “đừng hard-code”

  • Tách config khỏi code (YAML/JSON).
  • Snapshot config đã chạy.
  • Override từ CLI để thử nghiệm nhanh.

Hydra: override config từ CLI


python train.py optimizer.lr=0.0005 model.hidden_dim=128 data.version=v3

Ý tưởng: 1 run = 1 config cụ thể, có thể tái lập.

model:
  _target_: torch.nn.Linear
  in_features: 10
  out_features: 2

  _target_: torch.optim.Adam

from hydra.utils import instantiate

AutoML/HPO: 3 guardrails

  • Giới hạn ngân sách: trials/time/cost.
  • Pruning/Early stopping.
  • Baseline + metric rõ ràng.

Training at scale: 3 “điểm đau”

  • I/O bottleneck (data loading).
  • Scheduling GPU/TPU và chi phí.
  • Checkpoint + resume khi job bị gián đoạn.

2.
Evaluation &
Model Registry

Câu hỏi: "Eval để làm gì?"

Mục tiêu
Eval để giảm rủi ro khi deploy, không chỉ để “điểm cao”.

Offline evaluation: những thứ thường có

  • Split đúng (đặc biệt nếu dữ liệu có thời gian).
  • Baselines: non-ML hoặc model đơn giản.
  • Slice metrics: theo nhóm/điều kiện quan trọng.
  • Calibration nếu đầu ra là xác suất.

Quality gates: 4 lớp kiểm tra

  1. Data tests: schema, missing, ranges, outliers.
  2. Training tests: loss không diverge, overfit cảnh báo.
  3. Model tests: metric > baseline, fairness/safety cơ bản.
  4. Serving tests: latency/memory/throughput.

Ví dụ: data tests (gợi ý)


checks:
  - column: age
    type: range
    min: 0
    max: 120
  - column: income
    type: missing_rate
    max: 0.02

Câu hỏi: "Model registry là gì?"

Định nghĩa
Registry là “nguồn sự thật” cho model version: ai tạo, khi nào, từ run/code/data nào, và trạng thái (staging/prod).

Registry nên lưu gì?

  • Model artifact (weights).
  • Signature (input/output schema).
  • Metadata: owner, tags, description.
  • Link tới data version + code commit.
  • Eval report + approvals.

Vì sao Docker quan trọng?

Rollout strategies (an toàn → rủi ro)

  • Shadow deployment
  • Canary release
  • A/B testing

Blue-Green deployment

Ý tưởng
2 môi trường giống nhau: Blue (đang phục vụ) và Green (phiên bản mới). Cutover = switch traffic. Rollback = switch lại.

Sơ đồ Blue-Green (khái niệm)

Blue
  • model v1
  • stable SLO (Service Level Objective – mục tiêu mức dịch vụ)
Green
  • model v2
  • smoke tests
Traffic switch
LB / API Gateway / Ingress
→
Cutover: 100% → Green
Rollback: 100% → Blue

Autoscaling: lưu ý cho ML serving

  • Warmup (load model) có thể chậm → readiness probe.
  • Cold start ảnh hưởng latency tail (P95/P99).
  • Cache/feature store/network dễ thành bottleneck mới.

4.
CI/CD
cho ML

Câu hỏi: "CI/CD cho ML khác gì?"

Điểm khác
Không chỉ deploy code, mà deploy cả model artifact và kiểm soát data drift.

CI triggers phổ biến

  • Pull request: lint/tests/build image.
  • Push main: build + deploy staging.
  • Tag/release: promote production.
  • Schedule/event: kick off retraining.

Ví dụ GitHub Actions (rút gọn)


name: ci
on:
  pull_request:
  push:
    branches: [main]
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: pip install -r requirements.txt
      - run: pytest -q
  build:
    needs: [test]
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: docker build -t mymodel:${{ github.sha }} .

Quality gates trong CI/CD

  • Code tests + security scanning.
  • Data tests + schema checks.
  • Model regression tests (metric không giảm quá ngưỡng).
  • Serving smoke tests (latency/memory).

Công cụ CI/CD phổ biến

  • GitHub Actions, GitLab CI, Jenkins, CircleCI
  • ArgoCD/Flux (GitOps)
  • AWS CodeBuild/CodePipeline, GCP Cloud Build

5.
Monitoring &
Retraining

Câu hỏi: "Monitor cái gì?"

Hai nhóm chính
Operational metrics (SRE) + ML metrics (quality/drift).

Chú giải nhanh: SRE, SLI/SLO/SLA

  • SRE (Site Reliability Engineering): thực hành/nhóm đảm bảo độ tin cậy hệ thống.
  • SLI (Service Level Indicator): chỉ số đo chất lượng dịch vụ (latency, error rate, throughput, uptime…).
  • SLO (Service Level Objective): mục tiêu cho SLI (ví dụ: p99 latency < 2s).
  • SLA (Service Level Agreement): cam kết với khách hàng; vi phạm có thể bồi thường/credit.

Trong học liệu (Ch.8), uptime thường được định nghĩa bằng điều kiện latency (median/p99) + error rate.

Drift: 3 khái niệm hay gặp

  • Covariate shift: \(P(X)\) đổi (phân phối đầu vào/feature thay đổi).
  • Label shift: \(P(Y)\) đổi (tần suất nhãn thay đổi).
  • Concept drift: \(P(Y|X)\) đổi (“cùng X” nhưng Y khác; quan hệ X→Y đổi).

  • Input: snapshot phân phối feature ở production theo cửa sổ thời gian (ví dụ 1h, 1d).
  • Baseline: phân phối “chuẩn” (training set hoặc production ổn định 7–30 ngày).
  • Output: drift score theo từng feature + tổng hợp + dashboard + alert.

Gợi ý từ học liệu (Ch.8): drift detection chỉ “có khác”, bước khó hơn là “khác có quan trọng không”.

1) Thu thập (collect) để đo drift

  • Log input features (sau preprocessing) kèm timestamp, model_version, data_version.
  • Chia theo slice: region, device, customer-tier… (drift thường chỉ xuất hiện ở 1 nhóm).
  • Window: hourly/daily; lưu “summary” thay vì raw nếu cần (quantiles, histogram bins).
  • Baseline window: training distribution hoặc “golden period” production (không incident).
Tối thiểu cần gì để tính?
Với mỗi feature: histogram/bins (hoặc quantiles) ở baseline và ở current window.

2) Cách tính drift cho numeric features

  • Summary stats: so sánh mean/median/std/quantiles. Ví dụ: \(\Delta \text{median} = |\tilde{x}_{cur}-\tilde{x}_{base}|\).
  • KS test (Kolmogorov–Smirnov): \(D=\sup_x |F_{cur}(x)-F_{base}(x)|\).
  • PSI (Population Stability Index): chia bins, \(PSI=\sum_i (p_i-q_i)\ln\frac{p_i}{q_i}\).
  • Wasserstein-1 (Earth Mover): đo “chi phí” chuyển distribution (tốt khi shift theo vị trí).

Trong thực tế: PSI/KS hay dùng vì triển khai đơn giản, dễ giải thích cho business.

Ngưỡng (threshold) hay dùng cho numeric drift

Chỉ số Warning Critical Ghi chú
PSI \(\ge 0.10\) \(\ge 0.20\) quy ước phổ biến; tuỳ domain
KS p-value \(\le 0.05\) \(\le 0.01\) kèm min sample size để tránh “false alarm”
Quantile shift \(\Delta Q_{0.5}\) vượt ngưỡng \(\Delta Q_{0.5}\) rất lớn ngưỡng theo đơn vị feature (domain)

Best practice: đặt ngưỡng theo lịch sử “bình thường”, và phân cấp warn/crit + duration (ví dụ 3 cửa sổ liên tiếp).

3) Cách tính drift cho categorical features

  • Frequency shift: so sánh phân phối tần suất theo category.
  • Chi-square test: kiểm tra 2 phân phối rời rạc có khác đáng kể không.
  • Jensen–Shannon divergence (JS): \(JS(P||Q)=\frac12 KL(P||M)+\frac12 KL(Q||M)\), \(M=\frac{P+Q}{2}\).
  • OOV rate (out-of-vocabulary): % category mới chưa thấy ở baseline.

Với category có tail dài: gộp “OTHER” và theo dõi OOV/OTHER rate.

4) Drift cho embeddings / high-dimensional

  • Reduce dim: PCA/UMAP → rồi áp KS/PSI trên các thành phần chính.
  • Distance shift: theo dõi \(\|e\|_2\), cosine similarity tới centroid baseline.
  • MMD (Maximum Mean Discrepancy): đo khác biệt 2 phân phối nhiều chiều (khó hơn, cần kernel).
Gợi ý dạy học
Bắt đầu với PCA + PSI/KS trên 5–20 components để sinh viên dễ hình dung.

5) Khi drift vượt ngưỡng: làm gì?

  • Triage: drift xảy ra ở feature nào? slice nào? thời điểm nào?
  • Loại trừ lỗi hệ thống: schema change, bug pipeline, missing tăng, join sai, sai model_version.
  • Đo impact: nếu có label → xem metric (AUC/F1) theo slice; nếu không → proxy (calibration/confidence).
  • Hành động: rollback/pause rollout, fix pipeline, cập nhật feature, hoặc kick retrain.

Theo Ch.8: “nhiều dashboard drift” thực ra do lỗi nội bộ hơn là thay đổi thế giới.

Monitoring checklist (tối thiểu)

  • Schema & missing rate.
  • Feature statistics theo thời gian.
  • Prediction distribution + confidence.
  • Latency P95/P99 (95/99% request nhanh hơn ngưỡng) + error rate.
  • Dashboards + alert policies.

Observability vs Monitoring

  • Monitoring: biết “có vấn đề”.
  • Observability: đủ tín hiệu để biết “vấn đề ở đâu”.
  • Logs/metrics/traces cần tags để slice.

Alerting: tránh “alert fatigue”

  • Alert phải actionable: ai nhận, làm gì, runbook ở đâu.
  • Threshold + duration để giảm false positives.
  • Escalation theo mức độ ảnh hưởng.

Câu hỏi: "Khi nào retrain?"

Các trigger điển hình
Time-based · Performance-based · Volume-based · Drift-based.

Retraining: stateless vs stateful

  • Stateless: train từ đầu mỗi lần (thường tốn compute + cần nhiều data).
  • Stateful: fine-tune / incremental từ checkpoint (cập nhật bằng dữ liệu mới, có thể nhanh hơn).
  • Cần lineage rõ để debug chuỗi version.

Test in production: 4 kỹ thuật

  • Shadow deployment
  • Canary release
  • A/B testing
  • Bandits (nâng cao)

Tooling monitoring phổ biến

  • Metrics: Prometheus, Grafana
  • Logs: ELK/EFK, Loki
  • Traces: OpenTelemetry, Jaeger
  • Cloud: CloudWatch, Cloud Monitoring
