1.
Nền tảng Software Architecture

Software architecture là gì?

Định nghĩa làm việc
Software architecture là cách tổ chức các thành phần phần mềm, các quan hệ giữa chúng, và các quyết định kiến trúc để đạt mục tiêu chất lượng của hệ thống.

Trong dự án AI, kiến trúc phải trả lời gì?

  • Model được gọi ở đâu? theo request realtime hay batch?
  • Dữ liệu đi qua những lớp nào trước khi vào model?
  • Khi model sai thì hệ thống xử lý an toàn thế nào?
  • Làm sao mở rộng khi traffic tăng đột biến?

Model-centric vs System-centric

Chất lượng hệ thống cần cân bằng

  • Latency, throughput, availability
  • Scalability và cost
  • Maintainability và deployability
  • Security, privacy, compliance
  • Observability và incident response

Tại sao “accuracy tăng” chưa chắc “system tốt hơn”?

  • Model tốt hơn nhưng chậm hơn → UX xấu.
  • Model tốt hơn nhưng cost inference quá cao.
  • Model tốt hơn nhưng khó giải thích → khó duyệt nghiệp vụ.
  • Model tốt hơn nhưng gây hiệu ứng “creepy” với người dùng.

2.
Monolithic vs Microservices

Monolithic architecture

  • Một codebase/deploy unit lớn, nhiều module bên trong.
  • Dễ bắt đầu nhanh khi team nhỏ, domain đơn giản.
  • Debug local thuận tiện, transaction nội bộ dễ.

Ưu điểm Monolith

  • Đơn giản về vận hành ban đầu.
  • Ít chi phí network giữa service.
  • Release ban đầu nhanh.
  • Phù hợp MVP/POC.

Hạn chế Monolith

  • Khó scale riêng từng phần nóng.
  • Deploy một phần cũng phải redeploy toàn app.
  • Coupling cao, dễ “spaghetti dependencies”.
  • Khó tách ownership khi team lớn dần.

Ví dụ Monolith trong dự án AI

  • Một backend Flask/Django chứa:
    • API user
    • business logic
    • feature preprocessing
    • model inference
  • Phù hợp khi QPS thấp, ít team, yêu cầu thay đổi nhanh.

Microservices architecture

  • Hệ thống tách thành nhiều service nhỏ theo bounded context.
  • Mỗi service có lifecycle/deploy độc lập.
  • Service giao tiếp qua API hoặc event bus.

Ưu điểm Microservices

  • Scale theo chiều ngang từng service.
  • Deploy độc lập, giảm blast radius.
  • Dễ tổ chức team ownership theo domain.
  • Cho phép polyglot tech stack.

Hạn chế Microservices

  • Tăng độ phức tạp vận hành (network, observability, CI/CD).
  • Distributed debugging khó hơn.
  • Data consistency và transaction phức tạp.
  • Yêu cầu kỷ luật kiến trúc cao.

Ví dụ Microservices cho AI product

  • `user-service`, `catalog-service`, `recommendation-service`
  • `feature-service`, `ranking-service`, `notification-service`
  • `monitoring-service`, `feedback-label-service`

Khi nào chọn Monolith?

  • Team < 6 người, domain còn thay đổi nhiều.
  • Sản phẩm ở giai đoạn tìm product-market fit.
  • Ops chưa đủ năng lực cho distributed systems.

Khi nào chuyển sang Microservices?

  • Traffic tăng mạnh, hotspot rõ ràng.
  • Release bottleneck do đụng nhau nhiều team.
  • Cần scale và deploy riêng model inference.
  • Cần isolation rủi ro cho thành phần critical.

Chiến lược chuyển đổi an toàn

  • Monolith-first, tách dần theo domain.
  • Dùng Strangler pattern để thay thế từng phần.
  • Không “big bang rewrite” nếu chưa có lý do kinh doanh rõ.

3.
RESTful API, gRPC, Event

RESTful API (HTTP/JSON)

  • Phổ biến, dễ dùng từ web/mobile/service.
  • Resource-oriented: GET/POST/PUT/DELETE.
  • Dễ debug qua browser/Postman/curl.

Ưu nhược điểm REST

  • Ưu: đơn giản, ecosystem rộng.
  • Nhược: overhead JSON lớn, latency cao hơn nhị phân.
  • Phù hợp: client-facing APIs, public APIs.

gRPC (HTTP/2 + Protobuf)

  • Nhanh, binary protocol, strongly typed contracts.
  • Hỗ trợ streaming hai chiều.
  • Phù hợp giao tiếp nội bộ service-to-service.

Ưu nhược điểm gRPC

  • Ưu: performance cao, contract-first.
  • Nhược: khó debug thủ công hơn REST.
  • Phù hợp: low-latency microservices, realtime backend.

REST vs gRPC: chọn thế nào?

Event-driven communication

  • Publisher phát sự kiện, subscriber xử lý bất đồng bộ.
  • Giảm coupling thời gian giữa các service.
  • Tăng khả năng scale theo throughput.

Queue vs Pub/Sub (thực hành)

  • Queue: một message thường được một consumer xử lý.
  • Pub/Sub: một event có thể fan-out cho nhiều consumer.
  • Ví dụ: RabbitMQ (queue), Kafka (event streaming/pub-sub).

Khi nào dùng async?

  • Tác vụ nặng: xử lý file, trích xuất đặc trưng, batch scoring.
  • Tác vụ không cần phản hồi tức thì cho người dùng.
  • Cần chống backpressure và spike traffic.

API versioning

  • Không phá vỡ client cũ khi đổi schema.
  • Version qua path/header hoặc contract registry.
  • Deprecation policy rõ ràng cho đội tích hợp.

Idempotency và retry

  • Request lặp lại không tạo side effects sai.
  • Cần cho mạng không ổn định, timeout, retry tự động.
  • Dùng idempotency key cho các thao tác create/payment/order.

4.
Thành phần hạ tầng cốt lõi

Load Balancing là gì?

  • Phân phối request tới nhiều instance backend.
  • Tăng availability, giảm điểm nghẽn.
  • Có thể L4 hoặc L7 tùy nhu cầu.

Công nghệ Load Balancer phổ biến

  • Cloud LB: AWS ALB/NLB, GCP Load Balancer, Azure Load Balancer.
  • Self-managed: NGINX, HAProxy, Traefik, Envoy.
  • K8s ingress controllers: NGINX Ingress, Traefik, Istio Gateway.

Thuật toán cân bằng tải phổ biến

  • Round-robin
  • Least connections
  • Weighted routing
  • Latency-aware / geo-aware routing

API Gateway

  • Điểm vào thống nhất cho nhiều service phía sau.
  • Chức năng: authN/authZ, rate limit, routing, transform, logging.
  • Giảm phức tạp cho client.

Công nghệ API Gateway phổ biến

  • Managed: AWS API Gateway, Apigee, Azure API Management.
  • Open source: Kong Gateway, KrakenD, Tyk, Ambassador.
  • Service mesh edge: Istio ingress gateway, Envoy-based gateway.

Caching (Redis và các lớp cache)

  • Giảm latency, giảm tải DB/backend.
  • Lớp cache: client cache, CDN cache, application cache, DB cache.
  • Redis thường dùng làm in-memory key-value store.

Công nghệ caching phổ biến

  • In-memory: Redis, Memcached.
  • HTTP cache: Varnish, NGINX cache.
  • Application cache frameworks: Caffeine (Java), cachetools (Python).
  • Distributed cache trên cloud: AWS ElastiCache, Azure Cache for Redis.

Cache patterns

  • Cache-aside
  • Read-through / Write-through
  • Write-behind
  • TTL + invalidation strategy

Event queue / streaming

  • RabbitMQ: queue-centric, routing mạnh qua exchange.
  • Kafka: log/event streaming throughput cao, retention tốt.
  • Dùng cho async processing, decoupling, replay.

Công nghệ queue/event phổ biến

  • Queue: RabbitMQ, ActiveMQ, AWS SQS.
  • Streaming: Kafka, Redpanda, AWS Kinesis, Google Pub/Sub.
  • Workflow/event orchestration: Temporal, Airflow, Argo Workflows.

CDN là gì?

  • Phân phối nội dung tĩnh qua edge nodes gần người dùng.
  • Giảm latency toàn cầu, giảm tải origin.
  • Rất quan trọng cho frontend assets, media, model docs.

Công nghệ CDN phổ biến

  • Cloudflare CDN
  • AWS CloudFront
  • Google Cloud CDN
  • Azure CDN
  • Fastly, Akamai (enterprise)

VPS và VPN (khái niệm thường gặp)

  • VPS (Virtual Private Server): máy chủ ảo riêng trên hạ tầng cloud/DC.
  • VPN (Virtual Private Network): kênh mạng riêng, mã hóa giữa các mạng.
  • Dùng để bảo vệ truy cập admin, kết nối private giữa môi trường.

Ví dụ công nghệ VPS/VPN

  • VPS: DigitalOcean Droplets, Linode, Vultr, EC2.
  • VPN self-host: WireGuard, OpenVPN.
  • Enterprise/managed: AWS Client VPN, Cloudflare Zero Trust.

Serverless (AWS Lambda)

  • Chạy function theo event, không quản lý server trực tiếp.
  • Trả phí theo mức sử dụng.
  • Phù hợp xử lý ngắn, event-driven, bursty workload.

Công nghệ serverless phổ biến

  • AWS Lambda
  • Google Cloud Functions / Cloud Run
  • Azure Functions
  • OpenFaaS, Knative (self-host/Kubernetes)

Khi nào không nên serverless?

  • Tác vụ chạy lâu hoặc cần GPU liên tục.
  • Cold start không phù hợp SLA p95/p99 thấp.
  • Stateful connection dài, yêu cầu control runtime sâu.

Monitoring và Observability

  • 3 trụ cột phổ biến: metrics, logs, traces.
  • Mục tiêu: phát hiện nhanh + khoanh vùng nhanh + sửa nhanh.
  • Hệ microservices cần distributed tracing theo request-id.

Công nghệ monitoring phổ biến

  • Metrics: Prometheus + Grafana, Datadog.
  • Logs: ELK/EFK stack, Loki.
  • Tracing/APM: OpenTelemetry, Jaeger, Zipkin, New Relic.
  • Alerting/on-call: Alertmanager, PagerDuty, Opsgenie.

Alerting đúng cách

  • Alert policy rõ ngưỡng và thời gian vi phạm.
  • Notification đúng nhóm chịu trách nhiệm.
  • Có runbook giảm alert fatigue.

5.
Database & Storage cho AI systems

Vai trò data layer trong kiến trúc

  • Lưu trạng thái nghiệp vụ.
  • Lưu log/sự kiện/feature/model artifacts.
  • Phục vụ cả OLTP và analytics/ML pipelines.

Relational DB (SQL)

  • Ví dụ: PostgreSQL, MySQL.
  • Mạnh về transaction, consistency, query quan hệ.
  • Phù hợp user/account/order/billing.

Ví dụ công nghệ relational phổ biến

  • PostgreSQL, MySQL, MariaDB.
  • Managed SQL: AWS RDS/Aurora, Cloud SQL, Azure SQL.
  • Analytical SQL engines: Snowflake, BigQuery, Redshift.

NoSQL

  • Document (MongoDB), key-value (Redis), wide-column, graph.
  • Linh hoạt schema, scale ngang tốt cho một số workload.
  • Phù hợp session, catalog, metadata lớn.

Ví dụ công nghệ NoSQL phổ biến

  • Document: MongoDB, Couchbase.
  • Key-value: Redis, DynamoDB.
  • Wide-column: Cassandra, HBase.
  • Graph: Neo4j, Amazon Neptune.

Realtime DB

  • Ví dụ: Firebase Realtime Database / Firestore.
  • Đồng bộ theo thời gian thực cho mobile/web.
  • Phù hợp chat, trạng thái online, collaboration.

Ví dụ công nghệ realtime backend

  • Firebase Realtime Database, Cloud Firestore.
  • Supabase Realtime.
  • Ably, Pusher (managed realtime messaging).
  • Socket.IO + Redis adapter (self-host).

Object storage (file)

  • Ví dụ: Amazon S3, Google Cloud Storage, MinIO.
  • Lưu media, dataset, model artifact, backup.
  • Chi phí tốt cho lưu trữ lớn, durability cao.

Ví dụ công nghệ file/object storage

  • Amazon S3, Google Cloud Storage, Azure Blob Storage.
  • MinIO (S3-compatible on-prem).
  • Lưu model artifacts: MLflow artifact store trên S3/GCS.

Data warehouse / lake / lakehouse

  • Warehouse: phân tích có cấu trúc, BI tốt.
  • Lake: linh hoạt dữ liệu thô, chi phí thấp.
  • Lakehouse: kết hợp ưu điểm hai bên.

OLTP vs OLAP

Tiêu chíOLTPOLAP
Mục tiêugiao dịch realtimephân tích/báo cáo
Khối lượng querynhiều query nhỏít query nặng
Độ trễ mong muốnmss-phút

Polyglot persistence

  • Mỗi dịch vụ chọn DB phù hợp workload của nó.
  • Không ép “một DB cho tất cả”.
  • Đổi lại: tăng độ phức tạp vận hành và nhất quán dữ liệu.

Data consistency trong hệ phân tán

  • Strong consistency vs eventual consistency.
  • Saga pattern cho transaction liên service.
  • Outbox pattern để đồng bộ DB và event queue.

Schema management

  • Schema versioning cho API, events, database.
  • Backward/forward compatibility cho message.
  • Migration có rollback plan.

Backup, restore, disaster recovery

  • RPO/RTO rõ ràng theo mức độ critical.
  • Backup định kỳ + test restore thường xuyên.
  • Multi-AZ/region nếu yêu cầu sẵn sàng cao.

Bảo mật dữ liệu

  • Mã hóa at-rest và in-transit.
  • Phân quyền tối thiểu (least privilege).
  • Secrets management, audit logs, key rotation.

Luồng request online inference

  1. Client gọi API qua gateway.
  2. Gateway xác thực + route tới recommendation service.
  3. Service lấy feature (cache/DB), gọi ranking model.
  4. Trả kết quả và ghi log để quan sát + học lại.

Luồng dữ liệu học lại mô hình

  1. Sự kiện người dùng đẩy vào Kafka/RabbitMQ.
  2. Job xử lý tạo training dataset/labels.
  3. Training + evaluation + model registry.
  4. Promote model mới qua canary/blue-green.

Nút thắt thường gặp và xử lý

  • p95 latency cao → thêm cache + tối ưu query + tách service nóng.
  • DB quá tải → read replica, partition, CQRS.
  • Queue backlog → scale consumers + backpressure control.

Pattern hỗ trợ độ tin cậy

  • Timeout, retry, circuit breaker
  • Bulkhead isolation
  • Rate limiting và load shedding
  • Graceful degradation (fallback rule-based)

Blue-Green cho software architecture

  • Deploy bản mới lên môi trường Green.
  • Switch traffic qua LB/Gateway.
  • Nếu lỗi: route lại Blue trong vài giây.
  • Phù hợp component critical API/inference.

Auto scaling trong kiến trúc dịch vụ

  • Scale app/inference service theo QPS, CPU, p95 latency.
  • Scale consumer theo queue lag.
  • Giới hạn max để kiểm soát cost.
  • Warm-up để giảm cold start.

Quan sát hệ thống theo lớp

  • Edge: CDN hit ratio, TLS errors
  • Gateway: auth failures, 4xx/5xx
  • Service: p95 latency, saturation
  • Data: query latency, replication lag
  • ML: prediction drift, quality proxy metrics

Security architecture checklist

  • Network segmentation (private subnets, VPN/bastion).
  • Zero-trust cho service-to-service (mTLS nếu cần).
  • Secrets vault và rotation.
  • Audit trail cho admin/model changes.

7.
Thiết kế theo giai đoạn trưởng thành

Giai đoạn 1: MVP

  • Monolith + 1 DB + 1 queue đơn giản.
  • Deploy nhanh, đo usage thật.
  • Ưu tiên feedback loop với người dùng.

Giai đoạn 2: Growth

  • Tách service nóng: auth, inference, billing.
  • Thêm API gateway, cache, monitoring chuẩn hóa.
  • Tăng độ tự động CI/CD.

Giai đoạn 3: Scale

  • Microservices rõ bounded context.
  • Event-driven cho nhiều luồng bất đồng bộ.
  • DR strategy, multi-region cho workload quan trọng.

Anti-patterns thường gặp

  • Tách microservices quá sớm.
  • Không có observability nhưng hệ phân tán rộng.
  • Đưa mọi thứ vào queue “cho dễ scale”.
  • Không version API/schema/event.

Checklist chọn kiến trúc

  1. Traffic profile hiện tại và 12 tháng tới?
  2. Độ critical của tính năng AI?
  3. Năng lực đội vận hành hiện tại?
  4. Cost ceiling và SLO mong muốn?
  5. Ràng buộc pháp lý/bảo mật?

Tổng kết

  • Software architecture cho AI phải theo system-wide view.
  • Monolith vs Microservices là bài toán thời điểm và năng lực vận hành.
  • REST/gRPC/event-driven nên kết hợp theo đúng use case.
  • Các “khối lego” quan trọng: LB, API Gateway, cache, queue, CDN, serverless, monitoring, data layer.
  • Kiến trúc tốt là kiến trúc cân bằng giữa latency, reliability, cost, tốc độ phát triển.
