Core Idea
Nhiều đội “biết làm đúng” nhưng không làm được vì infrastructure chưa sẵn sàng. Hạ tầng tốt sẽ giảm lỗi, tăng tốc phát triển, và mở ra use case mới.

1.
Tổng quan Infrastructure cho AI

Vì sao AI infrastructure quan trọng?

  • ML systems phức tạp hơn app CRUD truyền thống.
  • Cần vận hành đồng thời data pipeline + training + serving + monitoring.
  • Tốc độ lặp mô hình phụ thuộc trực tiếp vào hạ tầng.

Infrastructure là gì trong ngữ cảnh ML?

  • Tập các “facility” nền tảng để phát triển và bảo trì hệ ML.
  • Bao gồm cả tài nguyên vật lý/ảo và công cụ quản trị vòng đời.
  • Mức đầu tư phụ thuộc quy mô và mức đặc thù bài toán.

Phổ nhu cầu hạ tầng

  • Đơn giản: notebook + Python + 1 máy.
  • Trung bình: nhiều use case, scale GB/TB, cần platform chung.
  • Đặc thù cao: tự xây hạ tầng chuyên biệt (search/autonomous).

4 lớp infrastructure điển hình (Ch.10)

  1. Storage & Compute
  2. Resource Management (scheduler/orchestrator)
  3. ML Platform (deployment/model store/feature store)
  4. Development Environment

Các quality attributes cần theo dõi

  • Performance (latency/throughput)
  • Reliability (SLO/uptime)
  • Scalability và elasticity
  • Security & compliance
  • Developer productivity

2.
Compute: CPU, GPU, TPU

Compute layer là gì?

  • Lớp tài nguyên thực thi workload: training, feature jobs, inference.
  • Ảnh hưởng trực tiếp tới tốc độ, chi phí và khả năng mở rộng.
  • Thường đến từ cloud compute (EC2/GCE/Azure VM).

Compute unit và các abstraction

  • Core/thread (mức máy)
  • Instance/VM (mức cloud)
  • Container/pod (mức orchestrator)
  • Job/task (mức workflow tools)

2 metric cơ bản của compute unit

  • Memory: bao nhiêu dữ liệu load được vào RAM/VRAM.
  • Operation speed: tốc độ thực thi phép tính.
  • Bên cạnh đó: I/O bandwidth cũng cực kỳ quan trọng.

CPU: điểm mạnh

  • Đa dụng, phù hợp backend logic và preprocessing.
  • Đơn giản triển khai, chi phí thấp hơn GPU.
  • Phù hợp inference nhẹ, batch ETL, orchestration.

GPU: điểm mạnh

  • Song song dữ liệu tốt cho tensor ops.
  • Hiệu quả cao cho deep learning training/inference.
  • Tối ưu với CUDA/cuDNN/TensorRT.

TPU: điểm mạnh

  • ASIC tối ưu cho tensor workloads quy mô lớn.
  • Hiệu năng/watt tốt trong một số bài toán.
  • Thường gắn chặt ecosystem cloud nhất định.

FLOPS và utilization

  • FLOPS chỉ là “khả năng lý thuyết”.
  • Hiệu năng thật phụ thuộc utilization + data pipeline.
  • Tối ưu đầu vào/I/O thường quan trọng như tối ưu model.

Ví dụ công nghệ compute phổ biến

  • CPU instances: AWS EC2 c/m/r series, GCP E2/N2
  • GPU instances: AWS g5/p4, GCP A2, Azure NC
  • TPU: Google Cloud TPU v4/v5
  • Distributed engines: Ray, Spark, Dask

3.
Storage: Disk và Cloud

Storage layer trong dự án AI

  • Nơi lưu data thô, data đã xử lý, model artifacts, logs.
  • Có thể tập trung 1 nơi hoặc phân tán đa hệ thống.
  • Thiết kế storage ảnh hưởng lớn tới cost và tốc độ.

Disk local: HDD vs SSD

  • HDD: rẻ, dung lượng lớn, IOPS thấp.
  • SSD/NVMe: nhanh, đắt hơn, hợp workload ngẫu nhiên.
  • Training cache cục bộ thường ưu tiên SSD/NVMe.

Object storage (cloud)

  • Lưu trữ file quy mô lớn, độ bền cao.
  • Phù hợp data lake, backup, model artifacts.
  • Đổi lại: latency cao hơn local disk/memory.

Hot/Warm/Cold tiers

  • Hot: truy cập liên tục, latency thấp, giá cao.
  • Warm: cân bằng chi phí và tần suất truy cập.
  • Cold/Archive: rẻ, truy xuất chậm.

Data formats và hiệu năng

  • Text formats: CSV/JSON dễ đọc, kém tối ưu.
  • Columnar/binary: Parquet/Avro tối ưu analytics.
  • Chọn format đúng có thể giảm mạnh chi phí truy vấn.

Storage locality và data loading

  • Dữ liệu càng xa compute thì latency càng cao.
  • Cân nhắc co-locate compute và storage theo vùng.
  • Dùng prefetch/caching để giảm bottleneck I/O.

Ví dụ công nghệ storage phổ biến

  • Object storage: AWS S3, GCS, Azure Blob, MinIO
  • Block storage: EBS, Persistent Disk
  • Warehouse/Lakehouse: BigQuery, Snowflake, Databricks
  • File system distributed: HDFS, CephFS

4.
Virtualization: VM và KVM

Virtualization là gì?

  • Tạo máy ảo chạy trên cùng phần cứng vật lý.
  • Tăng hiệu suất sử dụng tài nguyên.
  • Cho phép cô lập workload và quản trị linh hoạt.

Hypervisor type 1 vs type 2

  • Type 1 (bare metal): hiệu năng tốt hơn, dùng datacenter.
  • Type 2 (hosted): chạy trên OS thường, hợp dev/test.
  • Cloud lớn chủ yếu dùng kiến trúc kiểu type 1.

KVM là gì?

  • KVM (Kernel-based Virtual Machine) là virtualization module của Linux.
  • Biến Linux kernel thành hypervisor.
  • Nhiều cloud/private cloud dựa trên KVM.

VM phù hợp cho bài toán nào?

  • Workload cần isolation mạnh.
  • Hệ thống legacy hoặc cần kernel riêng.
  • Khi chưa cần tốc độ startup rất nhanh.

5.
Containers, Docker, Kubernetes

Vì sao cần container?

  • Môi trường production thường stateless và thay đổi liên tục.
  • Cần cách tái tạo môi trường nhất quán trên mọi máy.
  • Container giải quyết “works on my machine”.

Dockerfile, image, container

  • Dockerfile: công thức dựng môi trường.
  • Image: kết quả build từ Dockerfile.
  • Container: instance đang chạy từ image.

Ví dụ Dockerfile tối giản


FROM python:3.11-slim
WORKDIR /app
COPY requirements.txt .
RUN pip install -r requirements.txt
COPY . .
CMD ["python", "serve.py"]

Container registry

  • Nơi lưu và phân phối images.
  • Giúp CI/CD pull/push image theo version.
  • Cần policy scan lỗ hổng trước deploy.

Ví dụ registry phổ biến

  • Docker Hub
  • AWS ECR
  • Google Artifact Registry
  • GitHub Container Registry

Container orchestration là gì?

  • Quản lý nhiều container trên nhiều host.
  • Tự động scheduling, scaling, self-healing.
  • Đảm bảo HA cho service production.

Kubernetes (K8s) cơ bản

  • Pod là đơn vị deploy nhỏ nhất.
  • Deployment quản số replica và rollout.
  • Service/Ingress để expose ra mạng.

K8s trong workload AI

  • Inference services scale theo traffic.
  • Batch jobs/training jobs qua operators.
  • Tách node pools CPU/GPU để tối ưu chi phí.

K8s managed services

  • AWS EKS
  • Google GKE
  • Azure AKS
  • OpenShift (enterprise platform)

Docker Compose vs Kubernetes

  • Compose: tốt cho local/dev, 1 host.
  • K8s: production đa host, scale lớn.
  • Nhiều đội dùng Compose cho dev, K8s cho prod.

Pitfalls khi dùng containers/K8s

  • Image quá lớn, startup chậm.
  • Không có resource limits → noisy neighbors.
  • Thiếu readiness/liveness probes.
  • Secret quản lý không an toàn.

6.
Resource Management

Bản chất workload ML

  • Repetitive jobs: retrain định kỳ, batch scoring, data prep.
  • Phụ thuộc giữa các bước (DAG dependencies).
  • Nên cần scheduler/orchestrator phù hợp.

Cron

  • Chạy script theo lịch cố định.
  • Đơn giản nhưng không xử lý dependency phức tạp.
  • Phù hợp job nhỏ, ít ràng buộc.

Scheduler

  • Hiểu DAG, dependency, retry, priority.
  • Có thể trigger theo time hoặc event.
  • Quản hàng đợi jobs và phân bổ tài nguyên.

Orchestrator

  • Quan tâm tới máy/cluster/replication/provisioning.
  • Bổ sung hoặc thu hồi tài nguyên theo tải.
  • Thường kết hợp chặt với scheduler.

Workflow tools cho data science

  • Airflow, Argo, Prefect, Dagster

  • Đội ngũ quen code-first hay config-first?
  • Cần chạy local + cloud liền mạch không?
  • K8s là bắt buộc hay optional?
  • Tích hợp sẵn cloud/data stack nào?

7.
Cloud và AI Infrastructure Models

Public cloud vs private data center

  • Cloud: khởi động nhanh, elastic, capex thấp.
  • Private DC: kiểm soát cao, có thể tối ưu cost dài hạn.
  • Nhiều doanh nghiệp chọn hybrid.

IaaS là gì?

  • Cung cấp máy ảo, mạng, lưu trữ ở mức hạ tầng.
  • Người dùng tự quản OS/runtime/app.
  • Linh hoạt cao, cần vận hành nhiều hơn.

PaaS là gì?

  • Cung cấp nền tảng chạy ứng dụng, giảm gánh vận hành.
  • Scale, networking, deploy được quản lý nhiều hơn.
  • Đổi lại linh hoạt thấp hơn IaaS.

SaaS là gì?

  • Dùng dịch vụ hoàn chỉnh qua web/API.
  • Ít vận hành nhất, time-to-value nhanh.
  • Phụ thuộc vendor và giới hạn tùy biến.

8.
Networking Fundamentals

Vì sao networking quan trọng cho AI infra?

  • Model serving là bài toán network-heavy.
  • Data movement thường là bottleneck chi phí và latency.
  • Bảo mật hạ tầng gắn chặt với thiết kế mạng.

IP address

  • Định danh thiết bị trong mạng.
  • IPv4/IPv6, private/public ranges.
  • NAT giúp map private ra public.

DNS

  • Phân giải tên miền thành IP.
  • Rất quan trọng cho service discovery và failover.
  • TTL ảnh hưởng tốc độ cập nhật route.

TCP vs UDP

  • TCP: tin cậy, có kiểm soát thứ tự, dùng cho HTTP.
  • UDP: nhanh, không đảm bảo đầy đủ, hợp realtime media.
  • Nhiều AI API production dùng TCP-based protocols.

HTTP/HTTPS

  • HTTP là giao thức ứng dụng phổ biến cho APIs.
  • HTTPS = HTTP + TLS mã hóa đường truyền.
  • HTTP/2 giúp multiplexing, giảm overhead.

Proxy và Reverse Proxy

  • Forward proxy: đại diện client.
  • Reverse proxy: đại diện server, route và bảo vệ backend.
  • Dùng cho TLS termination, caching, rate limiting.

Load Balancer

  • Phân phối request nhiều backend nodes.
  • Hỗ trợ health checks và failover.
  • Kết hợp autoscaling để giữ SLO.

CDN

  • Đưa nội dung đến edge gần người dùng.
  • Giảm RTT và tải cho origin.
  • Hữu ích cho static assets và model-related artifacts.

9.
Dev Environment & Tooling

Dev environment gồm những gì?

  • IDE/notebook
  • Versioning cho code/data/experiments
  • CI/CD test suite
  • Kết nối an toàn tới môi trường staging/prod

IDE và notebook

  • IDE: VS Code, PyCharm, Vim, IntelliJ...
  • Notebook: Jupyter, Colab, SageMaker Studio.
  • Notebook mạnh cho khám phá nhưng dễ giảm tái lập nếu chạy cell không thứ tự.

Ưu/nhược của notebook stateful

  • Ưu: chạy lại từng bước nhanh, tiện EDA.
  • Nhược: khó tái lập nếu state ngầm.
  • Khuyến nghị: convert flow chính thành script/pipeline.

Standardize dev environments

  • Pin phiên bản Python và dependencies.
  • Dùng lockfile/conda env/poetry.
  • Giảm bug kiểu “chạy máy A được, máy B lỗi”.

Versioning stack điển hình

  • Code: Git/GitHub/GitLab
  • Data: DVC/LakeFS (tuỳ dự án)
  • Experiments: MLflow, W&B, Comet
  • Artifacts: S3/GCS + model store

CI/CD cho dev environment

  • PR checks: lint, tests, security scan.
  • Build container reproducible.
  • Deploy staging trước production.
  • Có rollback và release notes rõ ràng.

Tooling examples

  • CI/CD: GitHub Actions, CircleCI, GitLab CI
  • Workflow: Airflow, Argo, Prefect, Dagster
  • MLOps stack: Kubeflow, Metaflow, MLflow
  • Infra as Code: Terraform, Pulumi, Ansible

10.
ML Platform Components

ML Platform là gì?

  • Bộ công cụ dùng chung cho nhiều ứng dụng ML.
  • Giảm trùng lặp hạ tầng giữa các team.
  • Tăng tốc delivery và consistency.

3 thành phần phổ biến

  • Deployment service
  • Model store
  • Feature store

Deployment services

  • Đóng gói model + expose endpoint online/batch.
  • Cần hỗ trợ test in production (shadow/canary/A-B).
  • Theo dõi metric và rollback thuận tiện.

Model store

  • Không chỉ lưu binary model.
  • Cần lưu artifacts để tái lập và debug.
  • Model lineage là bài toán khó trong thực tế.

Artifacts nên lưu trong model store

  • Model definition + parameters
  • Featurize/predict functions
  • Dependencies/container
  • Data pointers/version
  • Code commit, experiment artifacts, owner tags

Feature store

  • Feature management (catalog, sharing, governance)
  • Feature computation (materialization)
  • Feature consistency giữa training và serving

Ví dụ công nghệ ML platform

  • Managed: SageMaker, Vertex AI, Azure ML
  • Open ecosystem: MLflow, Feast, Seldon, Ray Serve
  • Feature catalog: DataHub, Amundsen

Lộ trình triển khai 3 giai đoạn

  • Stage 1: cloud-first, containerize, metrics tối thiểu.
  • Stage 2: scheduler + model store + feature consistency.
  • Stage 3: optimize cost, autoscaling tinh chỉnh, governance.

Thực hành tốt cốt lõi

  • Standardize dev env trước khi “platform hóa” lớn.
  • Containerize sớm, đo benchmark thật, không chỉ spec.
  • Theo dõi cả cost và quality metrics song song.
  • Xây “small paved road” để team dùng dễ.

Tổng kết

  • Infrastructure tốt là nền tảng của MLOps tốt.
  • Compute và storage là lớp gốc; resource management quyết định hiệu quả vận hành.
  • Containers + Kubernetes giúp chuẩn hóa từ dev sang prod.
  • Cloud cho tốc độ, private cho kiểm soát; đa số chọn hybrid theo giai đoạn.
  • Networking và dev tooling là “điểm nghẽn vô hình” nhưng tác động rất lớn.
  • ML platform giúp tái sử dụng năng lực giữa nhiều use case AI.

  3. Đề xuất kế hoạch tối ưu cloud bill trong 6 tháng.
