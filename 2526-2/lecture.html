Động lực MLOps &

Câu hỏi: "MLOps là gì?"

Định nghĩa làm việc
MLOps là tập thực hành để tự động hoá và vận hành vòng đời ML: data → train → eval → release → deploy → monitor → update.

Vấn đề #1: ML có thể “fail silently”

  • Không crash, không exception — nhưng dự đoán sai.
  • Người dùng khó phát hiện nếu không có đo lường.
  • → Monitoring và test-in-production là bắt buộc.

Vấn đề #2: Dữ liệu không đứng yên

  • Training data và production data khác nhau theo thời gian.
  • Model “rot”: chất lượng giảm dần/sai theo phân khúc.
  • → Hệ thống phải hỗ trợ update liên tục.

Vấn đề #3: Nhiều thành phần → khó debug

  • Code + data + labels + features + configs + infra.
  • Một lỗi nhỏ ở pipeline tạo model “hợp lệ” nhưng sai.
  • → Cần lineage: model đến từ run/code/data nào.

DevOps vs MLOps

Khía cạnh DevOps MLOps
Artifact binary model + code + data snapshot
Test deterministic statistical + tolerances
Thay đổi code code + data + model
Quan sát logs/metrics + drift + model quality

Câu hỏi: "Production là gì?"

Một phổ (spectrum)
Từ "notebook cho báo cáo" → đến "dịch vụ realtime 99.9% uptime".
  • MLOps cần thiết khi có SLA/latency, update thường xuyên, nhiều model.
  • Nếu đơn giản: vẫn cần versioning + tracking tối thiểu.

Ba thói quen “tối thiểu”

  • Log mọi run: config + metrics + artifacts.
  • Version hóa: code/data/model/environment.
  • Đặt gates: không promote nếu chưa qua checks tối thiểu.

1.
Training &
Experiment Tracking

Câu hỏi: "Vì sao tracking?"

Mục tiêu
Khi có sự cố / cần cải tiến, trả lời nhanh: model này train bằng data nào, config nào, code nào.

Những thứ cần log cho mỗi run

  • Params: lr, batch size, seed…
  • Metrics: loss/accuracy/AUC, latency, cost…
  • Artifacts: checkpoint, plots, reports…
  • Environment: Python/lib versions, CUDA…

Quản lý cấu hình: “đừng hard-code”

  • Tách config khỏi code (YAML/JSON).
  • Snapshot config đã chạy.
  • Override từ CLI để thử nghiệm nhanh.

Hydra: override config từ CLI


python train.py optimizer.lr=0.0005 model.hidden_dim=128 data.version=v3

Ý tưởng: 1 run = 1 config cụ thể, có thể tái lập.

Hydra: instantiate từ config


model:
  _target_: torch.nn.Linear
  in_features: 10
  out_features: 2
optimizer:
  _target_: torch.optim.Adam
  lr: 0.001

AutoML/HPO: 3 guardrails

  • Giới hạn ngân sách: trials/time/cost.
  • Pruning/Early stopping.
  • Baseline + metric rõ ràng.

Training at scale: 3 “điểm đau”

  • I/O bottleneck (data loading).
  • Scheduling GPU/TPU và chi phí.
  • Checkpoint + resume khi job bị gián đoạn.

2.
Evaluation &
Model Registry

Câu hỏi: "Eval để làm gì?"

Mục tiêu
Eval để giảm rủi ro khi deploy, không chỉ để “điểm cao”.

Offline evaluation: những thứ thường có

  • Split đúng (đặc biệt nếu dữ liệu có thời gian).
  • Baselines: non-ML hoặc model đơn giản.
  • Slice metrics: theo nhóm/điều kiện quan trọng.
  • Calibration nếu đầu ra là xác suất.

Quality gates: 4 lớp kiểm tra

  1. Data tests: schema, missing, ranges, outliers.
  2. Training tests: loss không diverge, overfit cảnh báo.
  3. Model tests: metric > baseline, fairness/safety cơ bản.
  4. Serving tests: latency/memory/throughput.

Ví dụ: data tests (gợi ý)


checks:
  - column: age
    type: range
    min: 0
    max: 120
  - column: income
    type: missing_rate
    max: 0.02

Câu hỏi: "Model registry là gì?"

Định nghĩa
Registry là “nguồn sự thật” cho model version: ai tạo, khi nào, từ run/code/data nào, và trạng thái (staging/prod).

Registry nên lưu gì?

  • Model artifact (weights).
  • Signature (input/output schema).
  • Metadata: owner, tags, description.
  • Link tới data version + code commit.
  • Eval report + approvals.

Rollout strategies (an toàn → rủi ro)

  • Shadow deployment
  • Canary release
  • A/B testing
  • Blue-Green deployment

Blue-Green deployment

Ý tưởng
2 môi trường giống nhau: Blue (đang phục vụ) và Green (phiên bản mới). Cutover = switch traffic. Rollback = switch lại.

Sơ đồ Blue-Green (khái niệm)

Blue
  • model v1
  • stable SLO
Green
  • model v2
  • smoke tests
Traffic switch
LB / API Gateway / Ingress
→
Cutover: 100% → Green
Rollback: 100% → Blue

Autoscaling: lưu ý cho ML serving

  • Warmup (load model) có thể chậm → readiness probe.
  • Cold start ảnh hưởng latency tail (P95/P99).
  • Cache/feature store/network dễ thành bottleneck mới.

4.
CI/CD
cho ML

Câu hỏi: "CI/CD cho ML khác gì?"

Điểm khác
Không chỉ deploy code, mà deploy cả model artifact và kiểm soát data drift.

CI triggers phổ biến

  • Pull request: lint/tests/build image.
  • Push main: build + deploy staging.
  • Tag/release: promote production.
  • Schedule/event: kick off retraining.

Ví dụ GitHub Actions (rút gọn)


name: ci
on:
  pull_request:
  push:
    branches: [main]
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: pip install -r requirements.txt
      - run: pytest -q
  build:
    needs: [test]
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: docker build -t mymodel:${{ github.sha }} .

Quality gates trong CI/CD

  • Code tests + security scanning.
  • Data tests + schema checks.
  • Model regression tests (metric không giảm quá ngưỡng).
  • Serving smoke tests (latency/memory).

Công cụ CI/CD phổ biến

  • GitHub Actions, GitLab CI, Jenkins, CircleCI
  • ArgoCD/Flux (GitOps)
  • AWS CodeBuild/CodePipeline, GCP Cloud Build

5.
Monitoring &
Retraining

Câu hỏi: "Monitor cái gì?"

Hai nhóm chính
Operational metrics (SRE) + ML metrics (quality/drift).

Drift: 3 khái niệm hay gặp

  • Covariate shift: \(P(X)\) đổi.
  • Label shift: \(P(Y)\) đổi.
  • Concept drift: \(P(Y|X)\) đổi.

Monitoring checklist (tối thiểu)

  • Schema & missing rate.
  • Feature statistics theo thời gian.
  • Prediction distribution + confidence.
  • Latency P95/P99 + error rate.
  • Dashboards + alert policies.

Observability vs Monitoring

  • Monitoring: biết “có vấn đề”.
  • Observability: đủ tín hiệu để biết “vấn đề ở đâu”.
  • Logs/metrics/traces cần tags để slice.

Alerting: tránh “alert fatigue”

  • Alert phải actionable: ai nhận, làm gì, runbook ở đâu.
  • Threshold + duration để giảm false positives.
  • Escalation theo mức độ ảnh hưởng.

Câu hỏi: "Khi nào retrain?"

Các trigger điển hình
Time-based · Performance-based · Volume-based · Drift-based.

Retraining: stateless vs stateful

  • Stateless: train từ đầu mỗi lần.
  • Stateful: fine-tune từ checkpoint.
  • Cần lineage rõ để debug chuỗi version.

Test in production: 4 kỹ thuật

  • Shadow deployment
  • Canary release
  • A/B testing
  • Bandits (nâng cao)

Tooling monitoring phổ biến

  • Metrics: Prometheus, Grafana
  • Logs: ELK/EFK, Loki
  • Traces: OpenTelemetry, Jaeger
  • Cloud: CloudWatch, Cloud Monitoring

Tổng kết — MLOps như một hệ thống

Động lực
fail silently · drift · hệ nhiều thành phần → cần lifecycle có kiểm soát.
Training & Tracking
Config hóa + log runs + artifacts → tái lập & so sánh.
Eval & Registry
Quality gates + registry/version → promote có căn cứ.
Deployment
Online/batch + Docker + rollout + autoscaling.
CI/CD
Triggers + tests + build/publish + GitOps/rollback.
Monitoring
Drift + SLO + alerting → retrain đúng lúc.

Bài tập gợi ý

  1. Thiết kế MLOps cho 1 bài toán realtime với canary rollout.
  2. Định nghĩa 10 checks tối thiểu cho data + model trước khi promote.
  3. Chọn stack tracking/registry/monitoring phù hợp cho team 5–10 người.

1.
MLOps overview & lifecycle

MLOps là gì? (định nghĩa làm việc)

Định nghĩa
MLOps là tập thực hành (practice) để tự động hoá và vận hành vòng đời ML: từ dữ liệu → huấn luyện → đánh giá → triển khai → giám sát → cập nhật.

Trọng tâm: “làm cho việc cập nhật mô hình nhanh và an toàn” giống tư duy DevOps.

Vì sao cần MLOps?

  • ML systems thường fail silently: code vẫn chạy, nhưng dự đoán sai.
  • Data phân phối thay đổi theo thời gian → model “rot”.
  • Hệ thống có nhiều thành phần: data/labels/features/code/infra/teams → khó debug.
  • Cần cơ chế: tái lập, quan sát, và release có kiểm soát.

DevOps vs MLOps (so sánh nhanh)

Tiêu chí DevOps MLOps
Đơn vị thay đổi code code + data + model
Test deterministic thống kê + tolerances
Quan sát logs/metrics/traces + drift, data quality, model metrics
“Nợ kỹ thuật” code debt + data debt + versioning debt

MLOps “bọc” trọn vòng đời: không có monitoring thì không có lifecycle.

“Production” là một phổ (spectrum)

  • Có team xem “production” là tạo plot trong notebook cho business.
  • Có team xem “production” là phục vụ hàng triệu request/ngày, 99.9% uptime.
  • MLOps cần thiết khi:
    • có SLA/latency/uptime
    • model cập nhật thường xuyên
    • có nhiều model (không chỉ 1 model)

Myth: “Chỉ deploy 1–2 model”

  • Thực tế: một ứng dụng có thể có rất nhiều model (ETA, pricing, fraud, churn…).
  • Multi-region/multi-market → model variant theo quốc gia/segment.
  • Hệ quả: cần registry + versioning + routing.

Myth: “Nếu không làm gì, model vẫn tốt”

  • ML systems chịu data distribution shifts.
  • Model thường tốt nhất ngay sau training và giảm dần theo thời gian.
  • Do đó câu hỏi đúng: “Bao lâu có thể update model?”

Câu hỏi nhanh

  1. MLOps trong dự án của bạn là “tool” hay là “vòng đời”?
  2. Model của bạn có cần update thường xuyên không? vì sao?
  3. Nếu có nhiều model/biến thể, bạn quản lý release thế nào?

2.
Training & experiment tracking

Training không chỉ là “fit()”

  • Trong thực tế, training = chạy nhiều lần với nhiều cấu hình.
  • Cần:
    • quản lý cấu hình (config)
    • tracking metrics + artifacts
    • versioning code + data + env

Quản lý cấu hình: vì sao quan trọng?

  • Tách cấu hình khỏi code (YAML/JSON).
  • Dễ thử nghiệm và tái hiện.
  • So sánh các phiên bản mô hình dựa trên cấu hình.

Hydra (config động)

  • Ghép nhiều file config theo cây (hierarchy).
  • Override từ command line (rất hợp cho experiments).
  • Tự lưu lại config của mỗi run để tái hiện.

Ví dụ Hydra override


python train.py optimizer.lr=0.0005 model.hidden_dim=128

Thực hành tốt: mỗi run lưu snapshot config + git commit id.

Experiment tracking là gì?

Khái niệm
Experiment tracking = theo dõi tiến trình & kết quả huấn luyện (loss, metrics, resource…).
Versioning = lưu đủ thông tin để có thể so sánh/khôi phục run về sau.

Nên track những gì trong training?

  • Loss curve (train + validation splits).
  • Metrics quan trọng (accuracy/F1/AUC/perplexity…).
  • Sample prediction + ground truth để sanity check.
  • Tốc độ (steps/s, tokens/s), tài nguyên (CPU/GPU, memory).
  • Hyperparameters động (learning rate schedule, gradient norms…).

MLflow Tracking

  • Log params, metrics, artifacts theo từng run.
  • Có UI để so sánh runs.
  • Có thể mở rộng sang model registry.

Ví dụ MLflow


import mlflow
mlflow.set_experiment("thu-nghiem-phan-loai")
with mlflow.start_run():
    mlflow.log_param("model", "RandomForest")
    mlflow.log_param("n_estimators", 100)
    mlflow.log_metric("accuracy", 0.92)
    mlflow.log_artifact("model.pkl")

Weights & Biases (W&B)

  • Dashboard trực quan realtime.
  • Hợp tác nhóm (share runs/reports).
  • Artifacts: versioning dataset/model theo lineage kiểu “mô hình nào từ data nào”.

Ví dụ W&B


import wandb
wandb.init(project="thu-nghiem-phan-loai")
wandb.config.learning_rate = 0.01
for epoch in range(1, 6):
    loss = 0.1 * epoch
    acc = 0.9 - 0.02 * epoch
    wandb.log({"epoch": epoch, "loss": loss, "accuracy": acc})
wandb.finish()

Versioning: vì sao khó?

  • ML = code + data. Code dễ version (Git), data khó version (quá lớn).
  • “Diff” cho data không rõ ràng (thay file? checksum? schema?).
  • Ràng buộc pháp lý (ví dụ GDPR: xoá dữ liệu theo yêu cầu).

Một lỗi kinh điển: “không tái hiện được run tốt”

  • Thay đổi nhỏ về code/hyperparams/data subset nhưng không ghi lại.
  • Kết quả “promising” không lặp lại được → mất thời gian và niềm tin.
  • Nguyên tắc: mọi run quan trọng phải trace được.

Debugging ML: 3 mẹo thực chiến

  • Start simple: bắt đầu đơn giản rồi tăng dần.
  • Overfit 1 batch: nếu không overfit được, có thể sai pipeline/implementation.
  • Set random seed: giảm nhiễu khi so sánh experiments.

AutoML (đặt vào pipeline như thế nào?)

  • Soft AutoML: hyperparameter tuning (random/grid/Bayesian).
  • Hard AutoML: NAS, learned optimizer (đắt, ít team làm).
  • Trong MLOps: HPO là một “stage” có budget + guardrails.

Hyperparameter tuning: 3 phương pháp

  • Grid search: exhaustive, đắt.
  • Random search: thường hiệu quả hơn grid khi không gian lớn.
  • Bayesian optimization: thông minh hơn, ít thử hơn.

HPO trong production: guardrails

  • Budget: số trial tối đa, tổng GPU-hours, giới hạn chi phí.
  • Early stopping: cắt trial kém (ASHA/Hyperband, pruning).
  • Constraints: latency/cost/size (không chỉ accuracy).
  • Repro: log seed + config + code/data snapshot.

Training at scale: 3 “đau điểm”

  • Data parallelism: stragglers, gradient staleness (sync vs async).
  • Batch size lớn: diminishing returns, cần scale learning rate cẩn thận.
  • Model/pipeline parallelism: phức tạp hệ thống, khó debug.

Training performance: checkpoints & mixed precision

  • Gradient checkpointing: đổi compute để tiết kiệm memory.
  • Mixed precision: giảm thời gian/chi phí training (phụ thuộc hardware).
  • Tracking: log GPU utilization để phát hiện bottlenecks.

Câu hỏi nhanh

  1. Run “tốt nhất” của bạn có tái hiện được sau 1 tháng không?
  2. Nếu log quá nhiều metrics, bạn sẽ bị hại ở đâu?
  3. AutoML/HPO nên có “budget” và “early stop” thế nào?

3.
Evaluation & model registry

Evaluation: metric “một con số” là chưa đủ

  • Metrics tự thân “không có nghĩa” nếu thiếu baseline.
  • Production cần thêm: robustness, fairness, calibration, sanity.
  • Offline eval tốt giúp giảm rủi ro, nhưng không thay thế monitoring.

5 loại baseline hay dùng

  • Random baseline
  • Simple heuristic
  • Zero rule (luôn đoán lớp phổ biến nhất)
  • Human baseline
  • Existing solution (rule system / vendor)

Perturbation / invariance / directional tests

  • Perturbation: thêm nhiễu nhỏ để xem model có bền không.
  • Invariance: thay thuộc tính nhạy cảm (giới tính/chủng tộc) → output không nên đổi.
  • Directional expectation: thay input theo hướng “hợp lý” → output cũng hợp lý.

Model calibration (vì sao quan trọng)

  • Dự đoán xác suất 70% nghĩa là: trong 100 lần nói 70%, đúng ~70 lần.
  • Quan trọng cho:
    • decision-making có ngưỡng
    • ranking/recommender (độ đa dạng vs độ chính xác)
    • triage trong y tế/hỗ trợ khách hàng

Từ evaluation đến “release”

  • Không phải model nào “điểm cao” cũng được deploy.
  • Cần điều kiện:
    • đạt quality gates (accuracy + robustness + latency/cost)
    • đủ metadata để audit (data/code/version)
    • đủ thông tin để rollback

Model registry là gì?

Khái niệm
Model registry (model store/registry) là nơi quản lý phiên bản model + metadata + trạng thái (staging/production/archived).

Registry lưu những gì? (gợi ý)

  • Model artifact (weights/format), signature (input/output).
  • Run metadata: metrics, hyperparams, git commit.
  • Data reference/version, schema, feature definitions.
  • Dependencies/environment (conda/pip, container image).
  • Owner, tags, approval, release notes.

Câu hỏi nhanh

  1. Baseline nào phù hợp nhất cho bài toán của bạn?
  2. “Đạt accuracy” nhưng không calibrated có nguy hiểm không?
  3. Registry tối thiểu cần lưu gì để rollback an toàn?

4.
Deployment: API & Docker

Deploy là gì?

  • Deploy = làm cho model chạy được và được dùng được bởi hệ thống khác.
  • Đưa model ra khỏi “dev environment” → staging/prod.
  • Deploy dễ nếu bỏ qua phần khó; phần khó là scale + SLO + on-call + rollback.

Online vs batch prediction

Batch (async) Online (sync)
Mục tiêu throughput cao latency thấp
Khi nào dùng không cần tức thì (recsys định kỳ) fraud, assistant, realtime
Rủi ro stale predictions timeout, cost spikes

3 chế độ online prediction (gợi ý)

  • Online chỉ dùng batch features (ví dụ embeddings precompute).
  • Online dùng batch + streaming features (streaming prediction).
  • Hybrid: cache precompute cho query phổ biến + online cho phần còn lại.

Một lỗi phổ biến: 2 pipeline train vs inference

  • Batch pipeline cho training, stream pipeline cho inference.
  • Thay đổi một bên không phản ánh bên kia → training-serving skew.
  • Giải pháp hướng hệ thống:
    • chia sẻ code/feature definitions
    • feature store / unified processing

Prediction service: API contract (tối thiểu)

  • Endpoint: POST /predict
  • Input schema: versioned + validated.
  • Output: prediction + (optional) confidence + model_version.
  • Logging: request id, latency, errors, (hashed) features, model version.

Ví dụ API (pseudo)


@app.post("/predict")
def predict(payload: dict):
    X = payload["X"]
    y = MODEL.predict(X)
    return {"y": y, "model_version": MODEL_VERSION}

Docker trong deployment (vai trò)

  • Đóng gói model + dependencies + runtime.
  • Giảm “works on my machine”.
  • Hỗ trợ CI/CD, rollback theo image tag.

Dockerfile tối thiểu (minh hoạ)


FROM python:3.11-slim
WORKDIR /app
COPY requirements.txt .
RUN pip install -r requirements.txt
COPY . .
CMD ["uvicorn", "app:app", "--host", "0.0.0.0", "--port", "8000"]

Trong thực tế: pin versions, non-root, healthcheck, observability.

Tối ưu inference: 4 kỹ thuật phổ biến

  • Quantization: 32-bit → 16/8-bit (giảm size, tăng tốc).
  • Pruning: bỏ trọng số/nhánh ít hữu dụng (tăng sparsity).
  • Knowledge distillation: teacher → student nhỏ hơn.
  • Low-rank / kiến trúc gọn: MobileNet, SqueezeNet…

Từ batch sang online (checklist)

  • Có pipeline gần real-time (streaming features nếu cần).
  • Model đủ nhanh theo latency budget (optimize/compress/hardware).
  • Cache/precompute cho phần “đắt” (embeddings, candidate generation).
  • Đảm bảo consistency features giữa train & serve.

Rollout strategies

  • Canary: đẩy 1% traffic → quan sát → tăng dần.
  • A/B test: so sánh 2 model song song.
  • Shadow: model mới nhận request nhưng không ảnh hưởng output user.
  • Rollback: quay lại model cũ dựa trên registry + routing.

Blue-Green deployment (xanh dương - xanh lá)

  • Giữ đồng thời 2 môi trường giống nhau: Blue (đang phục vụ) và Green (phiên bản mới).
  • Kiểm thử Green trước khi chuyển traffic.
  • Switch traffic gần như tức thời qua load balancer/ingress.
  • Nếu lỗi: chuyển ngược về Blue rất nhanh.

  • Trước switch: 100% traffic vào Blue.
  • Sau switch: 100% traffic vào Green, Blue giữ làm fallback.

Khi nào dùng Blue-Green?

  • Yêu cầu downtime gần 0 và rollback cực nhanh.
  • Phù hợp release “all-at-once” hơn canary.
  • Đánh đổi: tốn tài nguyên hơn vì phải duy trì 2 môi trường.
  • Với DB migration cần backward-compatible để tránh lỗi khi rollback.

Auto Scaling cho dịch vụ ML

  • Horizontal scaling: tăng/giảm số replica theo tải.
  • Vertical scaling: tăng/giảm CPU/RAM/GPU cho pod/instance.
  • Signal thường dùng: CPU, memory, QPS, queue length, p95 latency.
  • Kết hợp min/max replicas + cooldown để tránh “scale flapping”.

Auto Scaling + MLOps: lưu ý thực chiến

  • Cold start model lớn có thể làm tăng latency đột ngột.
  • Dùng warm pool, pre-load model, hoặc tách service nặng/nhẹ.
  • Thiết kế SLO theo p95/p99 để trigger scale đúng lúc.
  • Theo dõi chi phí: scale tốt phải cân bằng latency và cloud bill.

Câu hỏi nhanh

  1. Bài toán của bạn phù hợp batch hay online?
  2. Deploy API cần log tối thiểu gì để debug “fail silently”?
  3. Canary vs A/B vs Shadow: chọn cái nào cho risk cao?

Quality gates (ví dụ)

  • Data checks pass (schema, nulls, ranges, drift bounds).
  • Offline metrics vượt baseline X%.
  • Latency/cost budget đạt.
  • Không vi phạm fairness/PII constraints (tuỳ dự án).

Testing trong ML pipeline: 4 lớp

  • Code tests: unit/integration (ETL, feature transforms).
  • Data tests: schema/constraints/distribution checks.
  • Model tests: offline metrics + regression tests (so với model cũ).
  • System tests: latency/load + end-to-end contracts.

Ví dụ data tests (gợi ý)

  • Schema: kiểu dữ liệu, cột bắt buộc, giá trị null rate tối đa.
  • Constraints: range, regex, uniqueness, referential integrity.
  • Freshness/volume: dữ liệu “đến” đúng giờ? số bản ghi có tụt bất thường?
  • Distribution: PSI/K-S theo feature quan trọng (theo ngày/tuần).

Reproducible environment

  • Pin dependencies (requirements/lockfile).
  • Container hoá training/inference nếu có thể.
  • Build trong CI để đảm bảo “khai báo đủ deps”.

Artefact lineage

  • Model version ↔ training code commit ↔ data version ↔ config snapshot.
  • Khi incident: trace từ prediction → model → pipeline → data.

6.
Monitoring & retraining

Vì sao ML monitoring khó?

  • Operational failures dễ thấy (timeout, 500…).
  • ML performance failures khó thấy vì thiếu/độ trễ ground truth labels.
  • Do đó ML systems thường “fail silently”.

Hai nhóm kỳ vọng (expectations)

  • Operational: latency, throughput, uptime, error rate.
  • ML performance: accuracy/F1/… hoặc business proxy metrics.

Violation của ML performance thường cần monitoring + feedback loop.

Monitoring checklist (tối thiểu)

  • Service: p95/p99 latency, error rate, saturation.
  • Data: schema drift, null rate, out-of-range, duplicates.
  • Prediction: distribution drift, calibration drift.
  • Business: conversion/CTR/cost per action (tuỳ bài toán).

Chiến lược “có label” trong production

  • Natural labels: click, purchase, report… (nhưng có bias).
  • Human review: sampling cho case rủi ro/edge cases.
  • Delayed labels: thiết kế dashboard theo lag (1h/1d/7d).
  • Golden set: tập benchmark cố định để so sánh phiên bản.

Alerting: tránh “bắn phá”

  • Phân cấp: warning vs critical; alert theo SLO, không theo “noise”.
  • Gắn context: model_version, data_version, feature offenders.
  • Rõ ownership: ai on-call? ai có quyền rollback?
  • Runbook: bước điều tra chuẩn hoá.

Nguyên nhân fail trong production

  • Software failures: dependency, deployment error, hardware, downtime…
  • ML-specific failures:
    • production data ≠ training data (train-serving skew)
    • data distribution shifts
    • edge cases
    • degenerate feedback loops

Data distribution shifts: 3 dạng (nhận diện)

  • Covariate shift: \(P(X)\) đổi, \(P(Y|X)\) giữ.
  • Label shift: \(P(Y)\) đổi, \(P(X|Y)\) giữ.
  • Concept drift: \(P(Y|X)\) đổi.

Trong vận hành: thường tập trung “làm gì để xử lý” hơn là phân loại học thuật.

Drift vs “bug nội bộ”

  • Nhiều dashboard drift thực ra do:
    • bug data pipeline
    • missing values sai
    • feature standardized bằng thống kê sai
    • deploy nhầm model version
  • Vì vậy monitoring cần kết hợp: data quality + lineage + versioning.

Drift detection: 2 hướng

  • Monitor metrics có label: accuracy/F1/AUC… (nếu có labels kịp thời).
  • Monitor distributions: feature stats (mean/quantiles), two-sample tests…

Statistical checks (thực dụng)

  • So sánh thống kê: min/max/mean/median/quantiles.
  • Hai-sample tests (K-S cho 1D; multivariate khó hơn).
  • Cảnh báo: “statistically significant” không luôn “practically important”.

Edge cases (vì sao quan trọng)

  • Model tốt 99.99% nhưng fail 0.01% có thể không dùng được nếu hậu quả nghiêm trọng.
  • Cần:
    • giám sát lỗi hiếm
    • guardrails không-ML
    • human-in-the-loop khi cần

Degenerate feedback loops

  • Output của hệ thống ảnh hưởng input tương lai → khuếch đại bias.
  • Ví dụ: recommender chỉ đẩy item “đang hot” → càng hot hơn.
  • Giảm thiểu:
    • randomization/exploration
    • positional features (giảm bias do vị trí hiển thị)

Retraining: khi nào kích hoạt?

  • Schedule-based: hằng ngày/tuần/tháng.
  • Drift-based: drift vượt ngưỡng.
  • Performance-based: metrics/business KPI giảm.
  • Event-based: đổi schema, thêm class mới, launch market mới.

“How often can we update?”

  • Tần suất retrain không chỉ là lựa chọn ML, mà là năng lực hệ thống.
  • Phụ thuộc:
    • data freshness + labeling latency
    • compute budget
    • automation maturity + CI/CD
    • risk tolerance + approval process

Testing in production

  • Offline eval không bắt hết vấn đề → cần test in prod.
  • Patterns:
    • shadow mode
    • canary
    • A/B testing

Incident playbook (tối thiểu)

  • Detect: alert drift/latency/errors.
  • Diagnose: versioning + logs + lineage.
  • Mitigate: fallback model, freeze updates, rollback.
  • Recover: backfill data, retrain, redeploy.
  • Prevent: thêm tests, tighten gates, improve observability.

Câu hỏi nhanh

  1. Nếu không có label trong production, bạn monitor bằng gì?
  2. Drift alert của bạn có thể là “false positive” do bug pipeline không?
  3. Retraining trigger nào an toàn nhất cho dự án risk cao?

Tổng kết

  • MLOps = vận hành vòng đời ML: train → deploy → monitor → update.
  • Experiment tracking + versioning là nền để tái lập và debug.
  • Evaluation cần baseline + robustness + calibration.
  • Deployment cần lựa chọn batch/online + rollout + rollback.
  • Monitoring phải bắt được fail silently: drift, edge cases, loops.
  • Retraining là năng lực hệ thống: “How often can we update?”

  1. Thiết kế lifecycle MLOps cho 1 dự án AI bạn chọn (RAG/RecSys/Fraud) và chỉ rõ artifacts cần version.
  2. Đề xuất 10 metrics/alerts cho monitoring (operational + ML) và cách điều tra khi alert xảy ra.
  3. Đề xuất policy promotion model: staging → canary → production, kèm rollback plan.
